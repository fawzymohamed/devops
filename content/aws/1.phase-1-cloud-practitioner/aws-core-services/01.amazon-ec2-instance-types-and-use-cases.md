# Amazon EC2 Instance Types & Use Cases

**Amazon EC2 (Elastic Compute Cloud)** gives you virtual servers in the cloud. Instead of buying physical hardware, you rent compute capacity from AWS and pay only for what you use. You can launch a server in minutes, scale it up when traffic spikes, and shut it down when you no longer need it.

## How EC2 Works

An EC2 **instance** is a virtual machine running on AWS hardware. When you launch an instance, you choose:

- **Instance type** -- how much CPU, memory, and network capacity you need
- **AMI (Amazon Machine Image)** -- the operating system and software to install
- **Storage** -- hard drives (EBS volumes) attached to the instance
- **Network** -- which VPC and subnet the instance runs in
- **Security group** -- firewall rules for inbound and outbound traffic

You can stop, start, and terminate instances at any time.

## EC2 Instance Families

AWS groups instances into **families** based on what they do best. Each family uses a letter code.

| Family | Letter Codes | Optimized For | Example Use Cases |
|--------|-------------|---------------|-------------------|
| **General Purpose** | T, M | Balanced CPU, memory, networking | Web servers, small databases, dev environments |
| **Compute Optimized** | C | High-performance processors | Batch processing, scientific modeling, gaming servers |
| **Memory Optimized** | R, X, z | Large amounts of RAM | In-memory caching (Redis), real-time analytics, SAP HANA |
| **Storage Optimized** | I, D, H | High sequential read/write to local storage | Data warehouses, distributed file systems, log processing |
| **Accelerated Computing** | P, G, Inf, Trn | GPU and custom hardware accelerators | Machine learning training, video encoding, 3D rendering |

### Reading Instance Names

Instance names follow a pattern: **family + generation + size**

```
m5.xlarge
│ │  └── Size (xlarge = 4 vCPUs, 16 GB RAM)
│ └──── Generation (5th generation)
└────── Family (M = General Purpose)
```

Common sizes from smallest to largest: `nano`, `micro`, `small`, `medium`, `large`, `xlarge`, `2xlarge`, `4xlarge`, and up.

### General Purpose (T, M)

Best for workloads that need a **balance** of compute, memory, and networking. The T family uses a **burstable** performance model -- the instance earns CPU credits when idle and spends them during traffic spikes.

**Use when:** You need a web server, small application, or development environment.

### Compute Optimized (C)

Built for tasks that need **raw CPU power**. These instances have the highest performing processors in their price range.

**Use when:** You run batch processing, scientific simulations, gaming servers, or high-performance web servers.

### Memory Optimized (R, X)

Designed for workloads that **process large datasets in memory**. These instances offer more RAM per dollar than other families.

**Use when:** You run in-memory databases, real-time big data analytics, or caching layers like Redis and Memcached.

### Storage Optimized (I, D, H)

Built for workloads that need **high sequential read/write access** to very large datasets on local storage.

**Use when:** You run data warehouses, distributed file systems (HDFS), or high-frequency online transaction processing (OLTP) systems.

### Accelerated Computing (P, G)

Include **GPUs or custom accelerators** for hardware-intensive tasks that a standard CPU cannot handle efficiently.

**Use when:** You train machine learning models, run graphics-intensive applications, or perform video transcoding.

## EC2 Pricing Models

AWS offers four main ways to pay for EC2 instances:

| Pricing Model | Commitment | Savings | Best For |
|---------------|-----------|---------|----------|
| **On-Demand** | None | 0% (full price) | Short-term, unpredictable workloads |
| **Reserved Instances** | 1 or 3 years | Up to 72% | Steady-state, predictable workloads |
| **Spot Instances** | None | Up to 90% | Fault-tolerant, flexible workloads |
| **Savings Plans** | 1 or 3 years | Up to 72% | Flexible commitment across instance types |

### On-Demand

Pay by the second (Linux) or by the hour (Windows) with no upfront costs. Start and stop anytime. This is the most flexible but most expensive option.

### Reserved Instances

Commit to 1 or 3 years for a specific instance type in a specific Region. You get a large discount in exchange for the commitment. Best for databases and applications that run 24/7.

### Spot Instances

Bid on unused EC2 capacity at steep discounts. AWS can reclaim your instance with a **2-minute warning** when demand rises. Perfect for batch jobs, data analysis, and any workload that can handle interruptions.

### Savings Plans

Commit to a consistent amount of compute usage (measured in dollars per hour) for 1 or 3 years. More flexible than Reserved Instances because you can change instance types, OS, or Region.

## Choosing the Right Instance

Ask these questions to pick the right instance:

1. **What does your workload need most?** CPU, memory, storage, or GPU?
2. **How predictable is your traffic?** Steady = Reserved. Spiky = On-Demand or Spot.
3. **Can your workload tolerate interruptions?** If yes, use Spot for savings.
4. **How long will you run this workload?** Months or years = Reserved or Savings Plans.

## Key Takeaways

- EC2 gives you virtual servers you can launch, scale, and terminate on demand
- Five instance families cover different workload types: General Purpose, Compute Optimized, Memory Optimized, Storage Optimized, and Accelerated Computing
- Instance names follow a pattern: family + generation + size (e.g., `m5.xlarge`)
- Four pricing models let you trade commitment for savings: On-Demand, Reserved, Spot, and Savings Plans
- Spot Instances offer the biggest savings (up to 90%) but AWS can reclaim them at any time
