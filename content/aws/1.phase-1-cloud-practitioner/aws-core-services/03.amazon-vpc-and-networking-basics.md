# Amazon VPC & Networking Basics

**Amazon VPC (Virtual Private Cloud)** lets you create your own isolated network inside AWS. Think of it as your private section of the AWS cloud where you control everything: IP addresses, subnets, routing, and who can access what. Every AWS resource you launch (EC2 instances, databases, Lambda functions) runs inside a VPC.

## VPC Core Concepts

When you create a VPC, you define a **CIDR block** -- a range of IP addresses. For example, `10.0.0.0/16` gives you 65,536 IP addresses to work with.

A VPC spans an entire **AWS Region** but you divide it into smaller pieces called **subnets**, each in a specific Availability Zone.

```
VPC: 10.0.0.0/16 (us-east-1)
├── Public Subnet:  10.0.1.0/24 (us-east-1a) -- 256 IPs
├── Public Subnet:  10.0.2.0/24 (us-east-1b) -- 256 IPs
├── Private Subnet: 10.0.3.0/24 (us-east-1a) -- 256 IPs
└── Private Subnet: 10.0.4.0/24 (us-east-1b) -- 256 IPs
```

## Public vs. Private Subnets

The difference between public and private subnets is about **internet access**.

### Public Subnet

A subnet is public when its route table sends internet-bound traffic to an **Internet Gateway (IGW)**. Resources here can have public IP addresses and communicate directly with the internet.

**Use for:** Web servers, load balancers, bastion hosts.

### Private Subnet

A subnet is private when it has **no route to an Internet Gateway**. Resources here cannot be reached from the internet directly.

**Use for:** Databases, application servers, internal services.

## Key Networking Components

### Internet Gateway (IGW)

An Internet Gateway connects your VPC to the internet. It handles traffic flowing both in and out. One IGW per VPC. Without it, nothing in your VPC can reach the internet.

### NAT Gateway

A **NAT (Network Address Translation) Gateway** lets resources in **private subnets** access the internet for outbound requests (like downloading software updates) while blocking inbound connections from the internet.

The NAT Gateway sits in a **public subnet** and has its own Elastic IP address. Private subnet route tables point internet-bound traffic to the NAT Gateway.

```
Private Instance → NAT Gateway (public subnet) → Internet Gateway → Internet
Internet → Internet Gateway → NAT Gateway → BLOCKED (no inbound route)
```

### Route Tables

Every subnet has a **route table** that tells traffic where to go. A route table contains rules (routes) that match destination IP ranges to targets.

| Destination | Target | Meaning |
|-------------|--------|---------|
| `10.0.0.0/16` | local | Traffic within the VPC stays local |
| `0.0.0.0/0` | igw-xxxxx | All other traffic goes to the Internet Gateway |

A private subnet's route table sends `0.0.0.0/0` to a NAT Gateway instead of an IGW, or has no default route at all.

### VPC Peering

**VPC Peering** connects two VPCs so they can communicate using private IP addresses. Traffic stays on the AWS network and never touches the public internet. Peering works across accounts and across Regions.

**Limitation:** VPC peering is not transitive. If VPC A peers with VPC B, and VPC B peers with VPC C, VPC A cannot talk to VPC C through B. You would need a direct peering connection between A and C.

## Security: Security Groups vs. NACLs

AWS gives you two layers of network security that work together.

| Feature | Security Group | Network ACL (NACL) |
|---------|---------------|-------------------|
| **Level** | Instance | Subnet |
| **Stateful/Stateless** | Stateful | Stateless |
| **Rules** | Allow only | Allow and Deny |
| **Default** | Denies all inbound, allows all outbound | Allows all inbound and outbound |
| **Evaluation** | All rules evaluated together | Rules evaluated in order by number |

### Security Groups

- **Stateful** means if you allow inbound traffic on port 80, the response traffic is automatically allowed out. You do not need a separate outbound rule.
- Attached to individual instances (or ENIs).
- Only **allow** rules -- you cannot create a deny rule. Traffic not matched by any allow rule is denied.

### Network ACLs

- **Stateless** means you must define both inbound and outbound rules. Allowing traffic in does not automatically allow the response out.
- Attached to **subnets** -- every resource in the subnet follows the same NACL.
- Support both **allow** and **deny** rules, evaluated in order by rule number (lowest number first).

**Best practice:** Use security groups as your primary defense. Use NACLs as an extra layer when you need to explicitly deny specific traffic (like blocking a known bad IP range).

## Key Takeaways

- A VPC is your isolated network in AWS where you launch all resources
- Subnets divide a VPC across Availability Zones; a route to an IGW makes a subnet public
- Internet Gateways connect your VPC to the internet; NAT Gateways let private resources make outbound internet requests
- Route tables control where network traffic goes within and outside the VPC
- Security groups are stateful firewalls at the instance level with allow-only rules
- NACLs are stateless firewalls at the subnet level with both allow and deny rules
- VPC peering connects two VPCs over private AWS networking but is not transitive
