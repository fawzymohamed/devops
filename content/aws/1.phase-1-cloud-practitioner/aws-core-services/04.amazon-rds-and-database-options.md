# Amazon RDS & Database Options

**Amazon RDS (Relational Database Service)** is a managed service that runs relational databases in the cloud. Instead of installing, patching, and backing up a database server yourself, RDS handles the heavy lifting. You focus on your data and application while AWS manages the infrastructure.

## Managed vs. Self-Managed Databases

You can run a database on an EC2 instance yourself (self-managed) or use RDS (managed). Here is what changes:

| Task | Self-Managed (EC2) | Managed (RDS) |
|------|-------------------|---------------|
| Hardware provisioning | You do it | AWS handles it |
| Database installation | You do it | AWS handles it |
| OS and DB patching | You do it | AWS handles it |
| Automated backups | You set it up | Built-in |
| High availability | You design it | One checkbox (Multi-AZ) |
| Scaling | You resize manually | Push-button scaling |
| Schema design | You do it | You do it |
| Query optimization | You do it | You do it |

**When to self-manage:** You need an unsupported engine, OS-level access, or a highly customized configuration.

**When to use RDS:** You want to offload infrastructure work and focus on your application.

## Supported Database Engines

RDS supports six relational database engines:

| Engine | Description | Use Cases |
|--------|-------------|-----------|
| **MySQL** | Popular open-source database | Web applications, CMS platforms, e-commerce |
| **PostgreSQL** | Advanced open-source database with rich features | Complex queries, geospatial data, analytics |
| **MariaDB** | MySQL-compatible fork with performance improvements | Drop-in MySQL replacement, high-throughput apps |
| **Oracle** | Enterprise commercial database | Legacy enterprise applications, financial systems |
| **SQL Server** | Microsoft's relational database | .NET applications, Windows-based workloads |
| **Amazon Aurora** | AWS-built engine compatible with MySQL and PostgreSQL | High-performance, cloud-native applications |

### Amazon Aurora

Aurora is AWS's own cloud-native database engine. It is compatible with MySQL and PostgreSQL, meaning your existing code works with Aurora without changes. Aurora offers:

- **5x throughput** of standard MySQL on RDS
- **3x throughput** of standard PostgreSQL on RDS
- Storage that automatically grows from 10 GB up to 128 TB
- 6 copies of your data across 3 Availability Zones
- Costs about 20% more than standard RDS but delivers significantly better performance

## Multi-AZ Deployments

**Multi-AZ** gives your database **high availability** by keeping a standby copy in a different Availability Zone. If the primary database fails (hardware issue, AZ outage), RDS automatically switches to the standby. This failover typically completes in 60-120 seconds.

```
Primary DB Instance (us-east-1a)
    │
    │ synchronous replication
    ▼
Standby DB Instance (us-east-1b) -- takes over if primary fails
```

Key points about Multi-AZ:

- **Synchronous replication** -- every write to the primary is immediately replicated to the standby
- **Automatic failover** -- no manual action needed, RDS updates the DNS record
- **No read traffic** -- the standby does not serve queries; it only waits for failover
- **Same Region** -- primary and standby must be in the same Region, different AZs

Multi-AZ is for **disaster recovery**, not performance improvement.

## Read Replicas

**Read Replicas** scale your database for **read-heavy workloads**. A Read Replica is a separate database instance that receives an asynchronous copy of data from the primary.

```
Application
    ├── Writes → Primary DB Instance
    └── Reads  → Read Replica 1
               → Read Replica 2
               → Read Replica 3
```

Key points about Read Replicas:

- **Asynchronous replication** -- there is a small delay (lag) between the primary and replicas
- **Read-only** -- replicas accept only SELECT queries, not INSERT/UPDATE/DELETE
- **Up to 15 replicas** for Aurora, up to 5 for other engines
- **Cross-Region** -- replicas can be in a different Region for lower latency or disaster recovery
- **Promotion** -- a Read Replica can be promoted to a standalone database (useful for migration or DR)

## Multi-AZ vs. Read Replicas

| Feature | Multi-AZ | Read Replicas |
|---------|----------|---------------|
| **Purpose** | High availability (failover) | Read scalability (performance) |
| **Replication** | Synchronous | Asynchronous |
| **Serves traffic** | No (standby only) | Yes (read queries) |
| **Failover** | Automatic | Manual promotion |
| **Cross-Region** | No (same Region) | Yes |

You can use both together: Multi-AZ for availability and Read Replicas for performance.

## RDS Backups and Storage

### Automated Backups

RDS takes daily snapshots of your database and stores transaction logs. You can restore to any point in time within your retention period (up to 35 days). Backups happen during a maintenance window you configure.

### Manual Snapshots

You can take snapshots at any time. Manual snapshots persist until you delete them, unlike automated backups which expire after the retention period.

### Storage Types

RDS offers three storage types:

- **General Purpose SSD (gp3)** -- balanced price and performance for most workloads
- **Provisioned IOPS SSD (io1/io2)** -- high-performance storage for I/O-intensive databases
- **Magnetic** -- legacy option, not recommended for new databases

## Key Takeaways

- RDS is a managed service that handles provisioning, patching, backups, and failover for relational databases
- Six engines are supported: MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Amazon Aurora
- Aurora is AWS's cloud-native engine offering up to 5x MySQL and 3x PostgreSQL performance
- Multi-AZ provides high availability through automatic failover to a standby in another AZ
- Read Replicas scale read-heavy workloads by offloading read queries to separate instances
- Multi-AZ is for disaster recovery (standby does not serve traffic); Read Replicas are for performance
- Automated backups and point-in-time restore protect your data with up to 35 days retention
