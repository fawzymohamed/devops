# AWS Lambda & Serverless Concepts

**AWS Lambda** lets you run code without managing servers. You upload your function, define what triggers it, and AWS handles everything else -- provisioning, scaling, patching, and monitoring. You pay only when your code runs, down to the millisecond.

## What is Serverless?

**Serverless** does not mean "no servers." Servers still exist, but you never see or manage them. AWS takes care of all infrastructure so you focus entirely on your code.

| Traditional Hosting | Serverless (Lambda) |
|--------------------|--------------------|
| You provision and manage servers | AWS manages all infrastructure |
| You pay for servers even when idle | You pay only when code executes |
| You handle scaling (add/remove servers) | AWS scales automatically to match demand |
| You patch the OS and runtime | AWS patches everything |
| Always running | Runs on demand, stops when done |

Serverless is ideal when you want to **run code in response to events** without worrying about infrastructure.

## How Lambda Works

A Lambda **function** is your code packaged and uploaded to AWS. It sits idle until an **event** triggers it. When triggered, AWS creates an **execution environment**, runs your code, returns the result, and then the environment goes idle or shuts down.

### Supported Runtimes

Lambda supports multiple programming languages:

- **Node.js** (JavaScript/TypeScript)
- **Python**
- **Java**
- **C# (.NET)**
- **Go**
- **Ruby**
- **Custom runtimes** (any language via the Runtime API)

### Event Sources and Triggers

Lambda functions are **event-driven**. Many AWS services can trigger a function:

| Trigger | Example Event |
|---------|---------------|
| **Amazon S3** | A file is uploaded to a bucket |
| **API Gateway** | An HTTP request hits your API endpoint |
| **Amazon SQS** | A message arrives in a queue |
| **DynamoDB Streams** | A record is inserted or updated in a table |
| **CloudWatch Events** | A scheduled cron job runs every hour |
| **Amazon SNS** | A notification is published to a topic |
| **Cognito** | A user signs up or signs in |

You can also invoke Lambda directly from your application using the AWS SDK.

## Lambda Pricing

Lambda pricing has two components:

1. **Requests** -- $0.20 per 1 million requests (first 1 million free each month)
2. **Duration** -- charged per GB-second (memory allocated x execution time)

**GB-second** means: if your function uses 1 GB of memory and runs for 1 second, that is 1 GB-second. If it uses 512 MB and runs for 2 seconds, that is also 1 GB-second.

**Free tier:** 1 million requests and 400,000 GB-seconds per month, every month, permanently.

**Example cost:** A function with 128 MB memory, running 200ms per invocation, called 10 million times per month costs roughly **$2.10**. The same workload on a small EC2 instance running 24/7 would cost ~$8-15/month.

## Lambda Limits

| Limit | Value |
|-------|-------|
| **Max execution time** | 15 minutes (900 seconds) |
| **Max memory** | 10,240 MB (10 GB) |
| **Deployment package (zipped)** | 50 MB (250 MB unzipped) |
| **Concurrent executions** | 1,000 per Region (can request increase) |
| **Temp storage (/tmp)** | 10,240 MB |

If your workload exceeds these limits, consider EC2, ECS, or breaking the work into smaller Lambda functions orchestrated by AWS Step Functions.

## Cold Starts

A **cold start** happens when Lambda creates a new execution environment for your function. This occurs when:

- The function is invoked for the first time
- The function has not been called for several minutes
- Traffic spikes and Lambda needs to create additional environments

During a cold start, AWS downloads your code, starts the runtime, and runs your initialization code. This adds **100ms to a few seconds** of latency depending on the runtime and package size.

**After the cold start**, Lambda reuses the same environment for subsequent invocations (a "warm" start), which is much faster.

**Tips to reduce cold starts:**

- Keep deployment packages small
- Use lighter runtimes (Node.js and Python start faster than Java)
- Use **Provisioned Concurrency** to keep environments warm (costs extra)

## Common Use Cases

| Use Case | How It Works |
|----------|-------------|
| **API backends** | API Gateway receives HTTP requests, triggers Lambda to process them |
| **File processing** | S3 upload triggers Lambda to resize images, parse CSVs, or transcode video |
| **Data transformation** | DynamoDB or Kinesis streams trigger Lambda to transform and load data |
| **Scheduled tasks** | CloudWatch Events triggers Lambda on a schedule (cron jobs) |
| **Chatbots and Alexa skills** | User input triggers Lambda to process and respond |
| **IoT backends** | IoT device events trigger Lambda for processing sensor data |

## Key Takeaways

- Serverless means AWS manages all infrastructure -- you just write and upload code
- Lambda runs code in response to events and scales automatically from zero to thousands of concurrent executions
- You pay per request and per execution duration (GB-seconds), with a generous free tier
- Lambda supports Node.js, Python, Java, C#, Go, Ruby, and custom runtimes
- Functions have a 15-minute maximum execution time and 10 GB maximum memory
- Cold starts add latency when a new execution environment is created; warm invocations are faster
- Common triggers include S3, API Gateway, SQS, DynamoDB Streams, and scheduled events
