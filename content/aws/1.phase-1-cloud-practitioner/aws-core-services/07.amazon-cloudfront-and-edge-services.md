# Amazon CloudFront & Edge Services

**Amazon CloudFront** is a **Content Delivery Network (CDN)** that delivers your content -- web pages, images, videos, APIs -- from locations close to your users. Instead of every request traveling to your origin server (which might be in a single Region), CloudFront caches copies at **edge locations** around the world and serves them from the nearest one.

## Why Use a CDN?

Without a CDN, a user in Tokyo requesting content from a server in Virginia must wait for data to travel across the Pacific Ocean and back. With CloudFront, the content is cached at an edge location in Tokyo and served instantly.

| Without CDN | With CloudFront |
|-------------|-----------------|
| Every request goes to the origin server | First request goes to origin; subsequent requests served from cache |
| High latency for distant users | Low latency -- content served from nearest edge |
| Origin handles all traffic | Edge locations offload most traffic from origin |
| Origin can become overloaded | Traffic is distributed globally |

## CloudFront Architecture

### Edge Locations

**Edge locations** are data centers around the world where CloudFront caches your content. AWS operates **450+ edge locations** across 90+ cities in 49 countries. These are separate from AWS Regions -- there are far more edge locations than Regions.

### Regional Edge Caches

Between edge locations and your origin, CloudFront maintains **Regional Edge Caches**. These are larger caches that hold content longer. When an edge location's cache expires, it checks the Regional Edge Cache before going all the way to the origin.

```
User → Edge Location (cache hit? serve it)
         │ cache miss
         ▼
      Regional Edge Cache (cache hit? serve it)
         │ cache miss
         ▼
      Origin Server (S3, ALB, custom)
```

### Origins

The **origin** is where CloudFront gets the original content. Supported origins include:

- **Amazon S3 bucket** -- for static files (images, CSS, JS, videos)
- **Application Load Balancer** -- for dynamic content from your web servers
- **Custom HTTP server** -- any server with a public URL, including on-premises servers
- **AWS MediaStore / MediaPackage** -- for video streaming

### Distributions

A **distribution** is a CloudFront configuration that defines:

- Which origin(s) to use
- How to cache content (TTL, headers, cookies)
- Which domain name to serve from (e.g., `d1234.cloudfront.net` or your custom domain)
- SSL/TLS certificate for HTTPS
- Geographic restrictions (block or allow specific countries)

CloudFront supports two distribution types:

- **Web distribution** -- for websites and APIs (most common)
- **RTMP distribution** -- for streaming media (deprecated; use web distribution with media origins instead)

## Caching Behavior

### How Caching Works

1. User requests `https://d1234.cloudfront.net/images/logo.png`
2. CloudFront routes the request to the nearest edge location
3. **Cache hit** -- the file is cached and served immediately
4. **Cache miss** -- CloudFront fetches the file from the origin, serves it, and caches it

### TTL (Time to Live)

**TTL** controls how long content stays in the cache. After the TTL expires, CloudFront checks the origin for a newer version on the next request.

| TTL Setting | Value | Use Case |
|-------------|-------|----------|
| **Default TTL** | 24 hours | General content |
| **Minimum TTL** | 0 seconds | Always check origin |
| **Maximum TTL** | 365 days | Content that rarely changes |

You can set TTL per file type. For example:
- `*.html` -- TTL of 1 hour (pages change often)
- `*.jpg` -- TTL of 30 days (images rarely change)
- `*.css` -- TTL of 7 days (stylesheets change occasionally)

### Cache Invalidation

When you update content at the origin but the cached version has not expired, you can **invalidate** the cache. Invalidation tells CloudFront to discard the cached copy so the next request fetches the latest version.

```bash
# Invalidate a specific file
aws cloudfront create-invalidation --distribution-id E1234 --paths "/images/logo.png"

# Invalidate all files in a directory
aws cloudfront create-invalidation --distribution-id E1234 --paths "/images/*"
```

**Tip:** A better approach for static assets is **cache busting** -- include a version or hash in the filename (e.g., `logo-v2.png` or `style.abc123.css`). This way the new file has a different URL and gets cached immediately without invalidation.

## CloudFront with S3

A common pattern is using CloudFront in front of an S3 bucket:

1. Store your files in an S3 bucket (images, CSS, JavaScript, videos)
2. Create a CloudFront distribution with the S3 bucket as the origin
3. Use an **Origin Access Control (OAC)** to restrict direct S3 access -- users must go through CloudFront
4. Users access files via the CloudFront URL with global caching

This keeps your S3 bucket private while serving content globally with low latency.

## CloudFront with ALB

For dynamic content (APIs, server-rendered pages):

1. Your application runs on EC2 instances behind an ALB
2. CloudFront sits in front of the ALB
3. CloudFront caches responses based on your TTL settings
4. Dynamic requests pass through to the ALB; static responses come from the cache

You can configure a single CloudFront distribution to use **multiple origins** and route by path:
- `/api/*` goes to the ALB origin
- `/static/*` goes to the S3 origin

## Key Takeaways

- CloudFront is a CDN that caches content at 450+ edge locations worldwide for low-latency delivery
- Edge locations are separate from AWS Regions and exist in more cities globally
- Origins can be S3 buckets, ALBs, or custom HTTP servers
- TTL controls how long content stays cached; cache invalidation forces immediate refresh
- Cache busting (versioned filenames) is preferred over invalidation for static assets
- Origin Access Control restricts S3 access so users must go through CloudFront
- A single distribution can route to multiple origins based on URL path patterns
