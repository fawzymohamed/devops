# Elastic Load Balancing & Auto Scaling

When your application grows, a single server cannot handle all the traffic. **Elastic Load Balancing (ELB)** distributes incoming requests across multiple servers, and **Auto Scaling** adds or removes servers based on demand. Together, they keep your application fast, available, and cost-efficient.

## What is Load Balancing?

A **load balancer** sits between users and your servers. It receives all incoming traffic and distributes it across multiple backend instances (called **targets**). This prevents any single server from becoming overloaded.

```
Users
  │
  ▼
Load Balancer
  ├── Instance 1
  ├── Instance 2
  └── Instance 3
```

Benefits:

- **High availability** -- if one instance fails, the load balancer sends traffic to healthy ones
- **Better performance** -- traffic is spread evenly so no single server is overwhelmed
- **Flexibility** -- add or remove instances behind the load balancer without downtime

## Load Balancer Types

AWS offers three active load balancer types. Each works at a different network layer.

| Type | Layer | Protocol | Best For |
|------|-------|----------|----------|
| **Application Load Balancer (ALB)** | Layer 7 (HTTP) | HTTP, HTTPS, gRPC | Web apps, microservices, APIs |
| **Network Load Balancer (NLB)** | Layer 4 (TCP/UDP) | TCP, UDP, TLS | Ultra-low latency, gaming, IoT |
| **Classic Load Balancer (CLB)** | Layer 4 & 7 | HTTP, TCP | Legacy applications (not recommended for new projects) |

### Application Load Balancer (ALB)

ALB operates at **Layer 7** (the application layer) and understands HTTP/HTTPS. It can make routing decisions based on the content of the request:

- **Path-based routing** -- `/api/*` goes to API servers, `/images/*` goes to image servers
- **Host-based routing** -- `api.example.com` goes to one target group, `www.example.com` goes to another
- **Query string routing** -- route based on URL parameters

ALB sends traffic to **target groups**, which are collections of instances, containers, or Lambda functions. Each target group has its own health check settings.

**Use ALB when:** You run HTTP-based applications and need content-based routing.

### Network Load Balancer (NLB)

NLB operates at **Layer 4** (the transport layer) and routes traffic based on IP address and port. It handles millions of requests per second with ultra-low latency.

- Does not inspect HTTP headers or content
- Supports static IP addresses and Elastic IPs
- Preserves the client's source IP address

**Use NLB when:** You need extreme performance, static IPs, or non-HTTP protocols (TCP, UDP).

### Classic Load Balancer (CLB)

The original AWS load balancer. It supports both Layer 4 and Layer 7 routing but lacks the advanced features of ALB and NLB. AWS recommends migrating existing CLBs to ALB or NLB.

## Health Checks

Load balancers send regular health check requests to each target. If a target fails the health check (returns an error or does not respond), the load balancer stops sending traffic to it.

| Setting | Description | Example |
|---------|-------------|---------|
| **Protocol** | HTTP, HTTPS, or TCP | HTTP |
| **Path** | URL to check | `/health` |
| **Interval** | Seconds between checks | 30 seconds |
| **Healthy threshold** | Consecutive successes to mark healthy | 3 |
| **Unhealthy threshold** | Consecutive failures to mark unhealthy | 2 |

## Auto Scaling

**Auto Scaling** automatically adjusts the number of EC2 instances based on demand. It ensures you have enough capacity during traffic spikes and reduces costs during quiet periods.

### Auto Scaling Groups (ASG)

An ASG defines:

- **Minimum capacity** -- the fewest instances you want running (e.g., 2)
- **Desired capacity** -- the number of instances Auto Scaling tries to maintain (e.g., 3)
- **Maximum capacity** -- the most instances allowed (e.g., 10)

```
Minimum: 2  |  Desired: 3  |  Maximum: 10

Low traffic:   [Instance 1] [Instance 2]
Normal traffic: [Instance 1] [Instance 2] [Instance 3]
High traffic:  [Instance 1] [Instance 2] [Instance 3] ... [Instance 10]
```

Auto Scaling also replaces unhealthy instances automatically. If a health check (from EC2 or the load balancer) reports an instance as unhealthy, Auto Scaling terminates it and launches a replacement.

### Launch Templates

A **launch template** tells Auto Scaling how to create new instances:

- AMI (operating system image)
- Instance type (e.g., `t3.medium`)
- Key pair for SSH access
- Security groups
- User data script (runs on startup)

### Scaling Policies

| Policy Type | How It Works | Example |
|-------------|-------------|---------|
| **Target Tracking** | Maintains a metric at a target value (like a thermostat) | Keep average CPU at 50% |
| **Step Scaling** | Adds/removes different amounts based on how far the metric is from the threshold | Add 2 instances if CPU > 70%, add 4 if CPU > 90% |
| **Simple Scaling** | Adds/removes a fixed number when a CloudWatch alarm triggers | Add 1 instance when CPU > 70% |
| **Scheduled Scaling** | Changes capacity at a specific time | Scale to 10 instances every weekday at 8 AM |

**Target tracking** is the easiest and most commonly recommended policy. You set a goal (e.g., 50% CPU utilization) and Auto Scaling figures out how many instances you need.

## ELB + Auto Scaling Together

These two services work as a team:

1. Auto Scaling launches new instances and registers them with the load balancer
2. The load balancer distributes traffic across all healthy instances
3. When demand drops, Auto Scaling removes instances and deregisters them
4. The load balancer health checks feed into Auto Scaling's health decisions

This combination gives you **automatic scaling** with **zero-downtime** traffic distribution.

## Key Takeaways

- Load balancers distribute traffic across multiple instances for availability and performance
- ALB (Layer 7) handles HTTP routing with path and host-based rules; NLB (Layer 4) handles ultra-low latency TCP/UDP traffic
- Health checks prevent the load balancer from sending traffic to failed instances
- Auto Scaling groups maintain a desired instance count and scale between min and max based on demand
- Target tracking scaling is the simplest policy -- set a target metric and let Auto Scaling adjust
- Auto Scaling automatically replaces unhealthy instances to maintain desired capacity
- ELB and Auto Scaling together provide automatic, elastic infrastructure that adjusts to traffic patterns
