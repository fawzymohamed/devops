# AWS Global Infrastructure (Regions, AZs, Edge Locations)

AWS runs a massive global network of data centers organized into **Regions**, **Availability Zones**, and **Edge Locations**. Understanding this structure is key to building reliable, fast, and compliant applications on AWS.

As of early 2025, AWS operates **39 Regions** with **123 Availability Zones** and plans to expand further.

## Regions

An AWS **Region** is a separate geographic area that contains multiple data centers. Each Region is completely independent from the others. Data does not leave a Region unless you explicitly move it.

**Examples:** `us-east-1` (N. Virginia), `eu-west-1` (Ireland), `ap-southeast-1` (Singapore)

### How to Choose a Region

Four factors guide your Region choice:

| Factor | Question to Ask |
|--------|----------------|
| **Compliance** | Does the law require data to stay in a specific country? |
| **Latency** | Where are your users? Pick the Region closest to them. |
| **Service availability** | Is the AWS service you need available in this Region? |
| **Pricing** | Prices differ by Region. Some are cheaper than others. |

**Compliance first.** If regulations say your data must stay in Germany, you use `eu-central-1` (Frankfurt) regardless of other factors.

## Availability Zones

An **Availability Zone (AZ)** is one or more physical data centers within a Region. Each AZ has its own power supply, cooling, and network connections. AZs within the same Region are connected by **high-speed, low-latency fiber** links.

Every Region has **at least three AZs**. Their physical separation protects against localized failures -- a power outage, flood, or fire in one AZ does not affect the others.

### How AZs Provide Fault Tolerance

When you deploy your application across multiple AZs, it stays running even if one AZ goes down.

**Example:** You run a web application with two EC2 instances -- one in AZ-1 and one in AZ-2. If AZ-1 has a power outage, the instance in AZ-2 keeps serving traffic. An Elastic Load Balancer routes users automatically to the healthy instance.

### AZ Naming

AWS names AZs by appending a letter to the Region code: `us-east-1a`, `us-east-1b`, `us-east-1c`. The letter-to-data-center mapping is randomized per AWS account to spread load evenly.

## Edge Locations

**Edge Locations** are smaller data centers spread across hundreds of cities worldwide. They sit much closer to end users than Regions do. AWS uses them primarily for **content caching and delivery**.

**Amazon CloudFront**, the AWS content delivery network (CDN), stores cached copies of your content at Edge Locations. When a user requests a file, CloudFront serves it from the nearest Edge Location instead of the origin server in a Region.

| Infrastructure | Count | Purpose |
|----------------|-------|---------|
| **Regions** | 39+ | Full AWS services, data storage |
| **Availability Zones** | 123+ | Fault tolerance within a Region |
| **Edge Locations** | 600+ | Content caching close to users |

### Other Edge Services

Besides CloudFront, Edge Locations also support:

- **AWS WAF** -- filters malicious web traffic at the edge
- **AWS Shield** -- DDoS protection at the edge
- **Route 53** -- DNS resolution at the edge
- **Lambda@Edge** -- runs lightweight functions at Edge Locations

## Local Zones and Wavelength Zones

**Local Zones** extend an AWS Region to a specific city. They place compute, storage, and database services closer to large population centers for single-digit millisecond latency.

**Wavelength Zones** embed AWS services inside 5G telecom networks for ultra-low latency mobile applications.

These are specialized options for latency-sensitive applications. For the Cloud Practitioner exam, focus on Regions, AZs, and Edge Locations.

## Key Takeaways

- A **Region** is an independent geographic area with multiple data centers
- Each Region has at least **three Availability Zones** for fault tolerance
- AZs are physically separated but connected by high-speed links
- Choose a Region based on compliance, latency, service availability, and cost
- **Edge Locations** cache content close to users via CloudFront for low latency
- Deploy across multiple AZs to keep your application running during failures
