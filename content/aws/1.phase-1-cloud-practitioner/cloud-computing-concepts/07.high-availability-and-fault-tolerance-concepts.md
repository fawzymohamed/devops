# High Availability & Fault Tolerance Concepts

Building applications that stay running during failures is a core cloud skill. AWS gives you the tools to design systems that recover quickly or keep working without interruption. The two key concepts are **high availability** and **fault tolerance**.

## High Availability vs. Fault Tolerance

These terms are related but different.

**High Availability (HA)** means your system stays operational **most of the time** with minimal downtime. If something breaks, the system recovers quickly -- but users may notice a brief interruption.

**Fault Tolerance (FT)** means your system keeps running **without any interruption** even when components fail. It requires full redundancy so a backup takes over instantly.

| Aspect | High Availability | Fault Tolerance |
|--------|------------------|-----------------|
| **Goal** | Minimize downtime | Zero downtime |
| **During failure** | Brief interruption possible | No interruption |
| **Cost** | Moderate | Higher (full redundancy) |
| **Example** | Auto Scaling replaces a failed server in ~2 minutes | Redundant database instantly fails over with no data loss |
| **Typical use** | Web applications, APIs | Financial systems, medical devices |

Most applications aim for **high availability**. Fault tolerance is reserved for critical systems where even seconds of downtime cause serious harm.

## Measuring Availability: The Nines

Availability is measured as a percentage of uptime per year. The industry uses "nines" as shorthand.

| Availability | Downtime per Year | Common Name |
|-------------|-------------------|-------------|
| 99% | 3.65 days | Two nines |
| 99.9% | 8.76 hours | Three nines |
| 99.99% | 52.56 minutes | Four nines |
| 99.999% | 5.26 minutes | Five nines |

Each additional nine is **much harder and more expensive** to achieve. Most production web applications target **99.9% to 99.99%**.

## RTO and RPO

Two metrics define your recovery goals:

**RTO (Recovery Time Objective)** -- how fast must you recover? This is the maximum time your application can be down after a failure.

**RPO (Recovery Point Objective)** -- how much data can you lose? This is the maximum acceptable data loss, measured in time.

| Metric | Question It Answers | Example |
|--------|-------------------|---------|
| **RTO** | How fast must we recover? | RTO of 1 hour = system must be back within 1 hour |
| **RPO** | How much data can we lose? | RPO of 15 minutes = backups every 15 minutes |

**Lower RTO and RPO = higher cost.** A system that recovers in seconds and loses zero data costs more than one that recovers in hours and accepts some data loss.

## AWS Services for High Availability

AWS provides managed services that handle much of the work needed for high availability.

| Service | How It Helps |
|---------|-------------|
| **Elastic Load Balancing** | Distributes traffic across multiple healthy instances in different AZs |
| **EC2 Auto Scaling** | Automatically adds or removes instances based on demand; replaces unhealthy ones |
| **Amazon RDS Multi-AZ** | Keeps a standby database replica in a second AZ with automatic failover |
| **Amazon S3** | Stores data across at least 3 AZs automatically with 99.999999999% durability |
| **Amazon Route 53** | DNS-level health checks and failover routing between endpoints |
| **Amazon CloudFront** | Serves cached content from Edge Locations even if the origin is down |

## Common Resilience Patterns

### Multi-AZ Deployment

Deploy your application across **two or more Availability Zones** in the same Region. An ELB sits in front and routes traffic only to healthy instances.

```
             ┌─────────────┐
             │   Users      │
             └──────┬───────┘
                    │
             ┌──────▼───────┐
             │     ELB      │
             └──┬───────┬───┘
                │       │
         ┌──────▼──┐ ┌──▼──────┐
         │  AZ-1   │ │  AZ-2   │
         │ EC2 + DB│ │ EC2 + DB│
         └─────────┘ └─────────┘
```

This is the **most common pattern** and the starting point for any production workload on AWS.

### Multi-Region Deployment

For applications that must survive an entire Region failure, deploy across **two or more Regions**. Route 53 handles DNS failover between Regions. This is more complex and expensive but achieves the highest level of resilience.

## Design Principles for Resilience

- **Assume everything fails** -- design for failure, not just success
- **Automate recovery** -- use Auto Scaling and health checks instead of manual intervention
- **Test your recovery** -- simulate failures regularly to verify your system recovers
- **Eliminate single points of failure** -- every critical component should have a backup
- **Scale horizontally** -- add more small instances instead of relying on one large one

## Key Takeaways

- **High availability** minimizes downtime; **fault tolerance** eliminates it
- Availability is measured in "nines" -- 99.99% allows about 52 minutes of downtime per year
- **RTO** is how fast you must recover; **RPO** is how much data loss is acceptable
- Multi-AZ deployment is the most common and cost-effective resilience pattern on AWS
- AWS services like ELB, Auto Scaling, RDS Multi-AZ, and S3 are built for high availability
- Always design for failure and automate recovery
