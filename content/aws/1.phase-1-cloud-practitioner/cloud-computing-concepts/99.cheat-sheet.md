# Cloud Computing Concepts - Quick Reference

## Cloud Computing Definition

**Cloud computing** = on-demand delivery of IT resources over the internet with pay-as-you-go pricing.

Three defining traits: **on-demand**, **over the internet**, **pay-as-you-go**.

---

## Six Benefits of Cloud Computing

| Benefit | What It Means |
|---------|---------------|
| **Trade upfront expense for variable expense** | No large hardware purchases; pay only for what you use |
| **Economies of scale** | Provider serves thousands of customers, so prices are lower |
| **Stop guessing capacity** | Scale up or down as needed instead of over/under-provisioning |
| **Increase speed and agility** | Launch resources in minutes, not weeks |
| **Stop spending money on data centers** | Focus on products, not hardware maintenance |
| **Go global in minutes** | Deploy to multiple Regions with a few clicks |

## Service Models (IaaS vs. PaaS vs. SaaS)

| Layer | On-Premises | IaaS | PaaS | SaaS |
|-------|------------|------|------|------|
| **Applications** | You | You | You | Provider |
| **Data** | You | You | You | You |
| **Runtime** | You | You | Provider | Provider |
| **Middleware** | You | You | Provider | Provider |
| **OS** | You | You | Provider | Provider |
| **Virtualization** | You | Provider | Provider | Provider |
| **Servers/Storage/Network** | You | Provider | Provider | Provider |

| Model | You Manage | AWS Examples | Use When |
|-------|-----------|-------------|----------|
| **IaaS** | OS, runtime, apps, data | EC2, S3, VPC | Need full control over environment |
| **PaaS** | App code, data | Elastic Beanstalk, Lambda, RDS | Want to focus on code, not servers |
| **SaaS** | Data and settings only | Gmail, Salesforce, Microsoft 365 | Need a ready-made application |

**Key pattern:** IaaS to SaaS = more provider control, less user control.

## Deployment Models

| Model | Description | Best For |
|-------|------------|----------|
| **Public** | Third-party provider, shared infrastructure (multi-tenant) | Startups, web apps, no strict data rules |
| **Private** | Dedicated to one organization (on-premises or hosted) | Government, healthcare, strict compliance |
| **Hybrid** | On-premises + public cloud connected together | Gradual migration, mixed compliance needs |
| **Multi-Cloud** | Two or more cloud providers simultaneously | Avoiding vendor lock-in, best-of-breed services |

| Feature | Public | Private | Hybrid | Multi-Cloud |
|---------|--------|---------|--------|-------------|
| **Cost** | Low | High | Medium | Medium-High |
| **Control** | Limited | Full | Mixed | Mixed |
| **Scalability** | Unlimited | Limited | High | High |
| **Complexity** | Low | Medium | High | Highest |

## AWS Global Infrastructure

| Component | Count | Purpose |
|-----------|-------|---------|
| **Regions** | 39+ | Full AWS services, data storage, independent geographic areas |
| **Availability Zones** | 123+ | Fault tolerance within a Region (min 3 per Region) |
| **Edge Locations** | 600+ | Content caching via CloudFront for low latency |

### Region Selection Factors

| Factor | Question |
|--------|----------|
| **Compliance** | Does law require data in a specific country? (Check first) |
| **Latency** | Where are your users? Pick the closest Region |
| **Service availability** | Is the service you need available in this Region? |
| **Pricing** | Prices vary by Region |

### AZ Key Facts

- Each AZ = one or more data centers with independent power, cooling, networking
- AZs within a Region connected by high-speed, low-latency fiber
- AZ naming: `us-east-1a`, `us-east-1b` (letter mapping randomized per account)

### Edge Services

CloudFront (CDN), AWS WAF, AWS Shield (DDoS), Route 53 (DNS), Lambda@Edge

## Well-Architected Framework -- 6 Pillars

| Pillar | Core Question | Key Principle |
|--------|--------------|---------------|
| **Operational Excellence** | How do we run and improve systems? | Perform operations as code; small, reversible changes |
| **Security** | How do we protect data and systems? | Least privilege; encrypt data in transit and at rest |
| **Reliability** | How do we recover from failures? | Auto-recover; scale horizontally; test recovery |
| **Performance Efficiency** | How do we use resources efficiently? | Use serverless; experiment with instance types |
| **Cost Optimization** | How do we avoid wasting money? | Pay only for what you use; analyze spending |
| **Sustainability** | How do we minimize environmental impact? | Maximize resource use; right-size instances |

**AWS Well-Architected Tool** = free console tool that reviews your architecture against the 6 pillars and generates a risk report.

## Cloud Economics & TCO

### CapEx vs. OpEx

| Model | Definition | IT Example |
|-------|-----------|------------|
| **CapEx** | Large upfront investment in physical assets | Buying servers, building a data center |
| **OpEx** | Pay-as-you-go for services consumed | Monthly AWS bill based on usage |

**Cloud = shift from CapEx to OpEx.** Lower financial risk, no upfront hardware commitment.

### On-Premises Hidden Costs

Hardware, software licenses, data center rent, power, cooling, physical security, staff, maintenance, downtime, end-of-life disposal.

**Typical finding:** Organizations save **30-50%** by migrating to the cloud.

### AWS Cost Tools

| Tool | Purpose |
|------|---------|
| **Pricing Calculator** | Estimate costs before migration |
| **Cost Explorer** | Visualize and analyze actual spending |
| **Budgets** | Set spending alerts and forecasts |
| **Cost and Usage Report** | Detailed line-item billing data |
| **Trusted Advisor** | Recommendations to reduce costs |

## High Availability & Fault Tolerance

### HA vs. FT

| Aspect | High Availability | Fault Tolerance |
|--------|------------------|-----------------|
| **Goal** | Minimize downtime | Zero downtime |
| **During failure** | Brief interruption possible | No interruption |
| **Cost** | Moderate | Higher (full redundancy) |
| **Use for** | Web apps, APIs | Financial systems, medical devices |

### The Nines

| Availability | Downtime/Year | Name |
|-------------|---------------|------|
| 99% | 3.65 days | Two nines |
| 99.9% | 8.76 hours | Three nines |
| 99.99% | 52.56 minutes | Four nines |
| 99.999% | 5.26 minutes | Five nines |

### RTO and RPO

| Metric | Measures | Example |
|--------|----------|---------|
| **RTO** (Recovery Time Objective) | How fast must you recover? | RTO of 1 hour = back up within 1 hour |
| **RPO** (Recovery Point Objective) | How much data can you lose? | RPO of 15 min = backups every 15 min |

**Lower RTO/RPO = higher cost.**

### AWS HA Services

| Service | How It Helps |
|---------|-------------|
| **Elastic Load Balancing** | Distributes traffic across healthy instances in multiple AZs |
| **EC2 Auto Scaling** | Adds/removes instances based on demand; replaces unhealthy ones |
| **RDS Multi-AZ** | Standby replica in another AZ with automatic failover |
| **Amazon S3** | Data stored across 3+ AZs; 99.999999999% durability |
| **Route 53** | DNS health checks and failover routing |

### Resilience Patterns

- **Multi-AZ:** Deploy across 2+ AZs in one Region (most common, cost-effective)
- **Multi-Region:** Deploy across 2+ Regions (highest resilience, highest complexity)

## Interview Quick Hits

| Question | Answer |
|----------|--------|
| What is cloud computing? | On-demand delivery of IT resources over the internet with pay-as-you-go pricing |
| What are the three cloud service models? | IaaS (you manage OS up), PaaS (you manage code/data), SaaS (you use the software) |
| Public vs. private vs. hybrid cloud? | Public = shared multi-tenant infrastructure. Private = dedicated to one org. Hybrid = on-premises + public cloud connected together |
| What is an Availability Zone? | One or more data centers with independent power/networking within a Region; each Region has at least 3 AZs |
| Name the 6 Well-Architected pillars. | Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability |
| What is the difference between CapEx and OpEx? | CapEx = large upfront investment (buying servers). OpEx = pay-as-you-go (monthly cloud bill). Cloud shifts IT from CapEx to OpEx |
| What is the difference between RTO and RPO? | RTO = maximum acceptable downtime after failure. RPO = maximum acceptable data loss measured in time |
| Why use Multi-AZ deployment? | Keeps your app running if one AZ fails; ELB routes traffic to healthy instances in other AZs. Most common AWS resilience pattern |
