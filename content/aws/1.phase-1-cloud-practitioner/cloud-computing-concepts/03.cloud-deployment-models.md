# Cloud Deployment Models (Public, Private, Hybrid, Multi-Cloud)

A **deployment model** defines where your cloud infrastructure lives and who controls it. Choosing the right model depends on your security needs, budget, compliance rules, and how much control you want.

## The Four Deployment Models

### Public Cloud

A **public cloud** is owned and operated by a third-party provider. Resources like servers, storage, and networking are shared across many customers using **multi-tenancy**. Each customer's data stays isolated, but the underlying hardware is shared.

**Examples:** AWS, Microsoft Azure, Google Cloud Platform

**Advantages:**
- No upfront hardware costs
- Scales on demand with no capacity limits
- Provider handles all maintenance and security of infrastructure

**Trade-offs:**
- Less control over the physical infrastructure
- Data leaves your premises (may concern some regulators)
- Shared infrastructure (though logically isolated)

### Private Cloud

A **private cloud** is dedicated to a single organization. No other customer shares the infrastructure. It can be hosted in your own data center or managed by a third party.

**Examples:** VMware private cloud, AWS Outposts (dedicated AWS hardware in your data center), OpenStack

**Advantages:**
- Full control over hardware, network, and security
- Meets strict regulatory and compliance needs
- Customizable to your exact requirements

**Trade-offs:**
- High upfront cost for hardware
- You handle maintenance, patching, and scaling
- Limited scalability compared to public cloud

### Hybrid Cloud

A **hybrid cloud** connects your on-premises infrastructure (or private cloud) with a public cloud. Workloads move between the two environments based on need.

This is the **most common enterprise strategy**. Organizations keep sensitive data on-premises while using the public cloud for scalable, less-sensitive workloads.

**Examples:** On-premises database + AWS for web tier, AWS Outposts + AWS cloud

**Advantages:**
- Keep sensitive workloads on-premises while scaling others in the cloud
- Flexibility to move workloads based on cost or performance
- Meets data residency and compliance requirements

**Trade-offs:**
- More complex to set up and manage
- Requires networking between environments (VPN, Direct Connect)
- Need staff skilled in both on-premises and cloud technologies

### Multi-Cloud

A **multi-cloud** strategy uses services from **two or more cloud providers** at the same time. For example, running your web application on AWS while using Azure for machine learning.

**Advantages:**
- Avoid vendor lock-in with any single provider
- Use the best service from each cloud (e.g., AWS for compute, GCP for ML)
- Increase resilience -- if one provider has an outage, others still work

**Trade-offs:**
- Highest complexity to manage
- Staff must learn multiple cloud platforms
- Data transfer between clouds costs money

## Quick Comparison

| Feature | Public | Private | Hybrid | Multi-Cloud |
|---------|--------|---------|--------|-------------|
| **Cost** | Low (pay-as-you-go) | High (own hardware) | Medium | Medium-High |
| **Control** | Limited | Full | Mixed | Mixed |
| **Scalability** | Unlimited | Limited | High | High |
| **Complexity** | Low | Medium | High | Highest |
| **Compliance** | Depends on provider | Full control | Flexible | Flexible |

## How to Choose

- **Public cloud** -- startups, web apps, and workloads without strict data rules
- **Private cloud** -- government, healthcare, or finance with strict compliance
- **Hybrid cloud** -- enterprises migrating gradually or with mixed compliance needs
- **Multi-cloud** -- organizations avoiding vendor lock-in or needing best-of-breed services

## Key Takeaways

- **Public cloud** shares infrastructure across customers -- low cost, high scalability
- **Private cloud** is dedicated to one organization -- full control, higher cost
- **Hybrid cloud** connects on-premises and public cloud -- flexibility with compliance
- **Multi-cloud** uses multiple providers -- avoids lock-in, highest complexity
- Most enterprises use a hybrid approach to balance security, compliance, and scalability
