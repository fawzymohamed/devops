# Data Protection & Encryption Basics

Data is the most valuable asset in your cloud. Protecting it means making sure that **even if someone gets access to the raw storage or intercepts the network traffic**, the data is unreadable without the encryption key.

Think of encryption like putting your valuables in a locked safe. Even if a thief breaks into your house, they cannot open the safe without the combination. Encryption turns readable data into scrambled text that only the right key can unlock.

| State | What It Protects | How It Works | AWS Tools |
|-------|-----------------|-------------|-----------|
| **At rest** | Data sitting on storage (S3, EBS, RDS) | Data is encrypted when written, decrypted when read | KMS, S3 encryption, EBS encryption |
| **In transit** | Data moving between your computer and AWS, or between AWS services | A secure tunnel wraps the data so nobody can read it | SSL/TLS, HTTPS, VPN |

## AWS Key Management Service (KMS)

**AWS KMS** is the central service for creating and managing **encryption keys**. Instead of handling raw encryption keys yourself, KMS stores them securely and lets you control who can use them through IAM policies.

### Types of KMS Keys

| Key Type | Managed By | Rotation | Cost | Use When |
|----------|-----------|----------|------|----------|
| **AWS owned keys** | AWS (fully hidden) | AWS handles it | Free | Default encryption on some services |
| **AWS managed keys** | AWS (visible in your account) | Automatic (yearly) | Free (per-use API costs) | Simple encryption with audit trail |
| **Customer managed keys** | You | Optional (you configure) | $1/month + API costs | Full control over key policy, rotation, and deletion |

### How KMS Works with Other Services

KMS integrates with many AWS services. When you enable encryption on S3, RDS, EBS, or Lambda, those services use KMS behind the scenes to encrypt and decrypt your data.

```
Your App  -->  Calls S3 PutObject  -->  S3 asks KMS for data key
                                          -->  KMS returns encrypted key
                                          -->  S3 encrypts your file
                                          -->  Stores encrypted file + encrypted key
```

You never handle the raw encryption key. KMS does the heavy lifting.

## S3 Encryption Options

Amazon S3 encrypts all new objects **by default** using SSE-S3. But you can choose different options based on your needs:

### Server-Side Encryption (AWS Encrypts for You)

| Option | Key Management | Cost | Best For |
|--------|---------------|------|----------|
| **SSE-S3** | S3 manages keys completely | Free | Default, simplest option |
| **SSE-KMS** | KMS manages keys, you control policies | KMS API costs | Audit trail, key rotation control |
| **SSE-C** | You provide the key with every request | Free (you manage keys) | Full key control, strict compliance |

### Client-Side Encryption (You Encrypt Before Upload)

With client-side encryption, you encrypt data **on your machine** before sending it to S3. AWS never sees the unencrypted data. You manage the entire encryption process and keys.

**Use when:** Regulations require that the cloud provider never has access to unencrypted data.

## SSL/TLS -- Encryption in Transit

**SSL/TLS** (Secure Sockets Layer / Transport Layer Security) creates an encrypted tunnel between two points. When you see `https://` in a URL, SSL/TLS is protecting that connection.

**How it works:**

1. Your browser connects to the AWS endpoint
2. The server presents a **certificate** to prove its identity
3. Both sides agree on an encryption method
4. All data flows through the encrypted tunnel

**AWS enforces HTTPS** on most of its API endpoints. When your application talks to S3, DynamoDB, or any AWS service, the connection uses TLS by default.

**AWS Certificate Manager (ACM)** lets you create and manage SSL/TLS certificates for your applications at no extra cost. You can attach certificates to load balancers, CloudFront distributions, and API Gateway endpoints.

## Key Takeaways

- **Encryption at rest** protects stored data; **encryption in transit** protects data on the network -- you need both
- **AWS KMS** is the central service for managing encryption keys with three key types: AWS owned, AWS managed, and customer managed
- S3 encrypts all new objects by default with **SSE-S3**; use **SSE-KMS** for audit trail and key control, **SSE-C** when you supply your own key
- **Client-side encryption** means you encrypt before sending to AWS -- the provider never sees plaintext
- **SSL/TLS** (HTTPS) protects data in transit and is enforced on AWS API endpoints by default
- AWS Certificate Manager provides free SSL/TLS certificates for your applications
