# Shared Responsibility Model

When you use AWS, security is **not** all on you, and it is **not** all on AWS. You share the work. The **Shared Responsibility Model** draws a clear line between what AWS secures and what you must secure yourself.

Think of it like renting an apartment. The landlord (AWS) maintains the building structure, plumbing, and electrical wiring. You (the customer) lock your doors, protect your belongings, and decide who gets a spare key.

### AWS Handles: Security OF the Cloud

AWS protects the **infrastructure** that runs all cloud services:

- **Physical security** -- data centers with biometric access, 24/7 guards, and cameras
- **Hardware** -- servers, storage drives, and networking equipment
- **Software** -- the hypervisor that isolates your virtual machines from others
- **Global network** -- fiber cables, routers, and edge locations connecting Regions
- **Managed service infrastructure** -- the servers behind S3, DynamoDB, Lambda, and RDS

You never touch this layer. AWS audits and certifies it through programs like SOC and ISO.

### You Handle: Security IN the Cloud

You protect **everything you put on AWS** and **how you configure it**:

- **Data** -- what you store and how you encrypt it
- **Identity and access** -- IAM users, roles, policies, and passwords
- **Operating system** -- patching the OS on EC2 instances
- **Network configuration** -- security groups, NACLs, and routing tables
- **Application security** -- your code, libraries, and dependencies
- **Encryption** -- choosing to encrypt data at rest and in transit

If your S3 bucket is public, that is your responsibility. AWS gave you the tools to lock it down -- using them is up to you.

## How Responsibility Shifts by Service Type

The amount of work you do depends on how managed the service is.

| Layer | EC2 (IaaS) | RDS (PaaS) | Lambda (Serverless) | S3 (Managed) |
|-------|-----------|-----------|-------------------|-------------|
| **Your data** | You | You | You | You |
| **Your code** | You | You | You | N/A |
| **IAM & access** | You | You | You | You |
| **OS patching** | You | AWS | AWS | AWS |
| **Runtime** | You | AWS | AWS | AWS |
| **Infrastructure** | AWS | AWS | AWS | AWS |

**Pattern:** The more managed the service, the less you manage. With EC2, you patch the OS yourself. With RDS, AWS patches the database engine. With Lambda, AWS manages everything except your code and permissions.

## A Simple Memory Trick

Ask yourself: **"Can I touch it in the AWS console or CLI?"**

- If **yes** -- it is probably your responsibility (security groups, IAM policies, encryption settings)
- If **no** -- it is probably AWS's responsibility (physical servers, hypervisor, data center locks)

## Key Takeaways

- AWS handles **security OF the cloud** -- physical infrastructure, hardware, networking, and the hypervisor
- You handle **security IN the cloud** -- your data, IAM, OS patches, encryption, and firewall rules
- Customer responsibility **decreases** as you move from IaaS (EC2) to managed/serverless services (Lambda, S3)
- If you can configure it in the AWS console, it is likely your responsibility to secure it
- The model applies to every AWS service -- always ask "who is responsible for this layer?"
