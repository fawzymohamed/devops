# IAM Users, Groups, Roles & Policies

Now that you know the IAM building blocks, let's dive deeper into how they work together. The goal: give the right people and services the right access, and nothing more.

## IAM Users

An IAM user represents a **person or application** that interacts with AWS. Each user gets a unique name within your account and can have two types of credentials:

- **Console password** -- for logging into the AWS Management Console (humans)
- **Access keys** -- a key ID and secret key for API/CLI access (applications and scripts)

**Best practices for IAM users:**

- Create **one user per person** -- never share accounts
- Assign permissions through **groups**, not directly on users
- **Rotate access keys** regularly and delete unused ones
- Enable **MFA** for every human user

## IAM Groups

Groups organize users who need the same permissions. Attach a policy to the group, and every user in that group gets those permissions automatically.

**Key rules about groups:**

- A user can belong to **multiple groups** (and gets all their combined permissions)
- Groups **cannot be nested** -- you cannot put a group inside another group
- Groups are for **organizing humans**, not services -- use roles for services

## IAM Policies in Detail

A policy is a **JSON document** that defines permissions. Here is the anatomy of a policy:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }
  ]
}
```

| Element | What It Does | Example Values |
|---------|-------------|----------------|
| **Version** | Policy language version (always use this date) | `"2012-10-17"` |
| **Effect** | Allow or Deny the actions | `"Allow"` or `"Deny"` |
| **Action** | Which API calls this applies to | `"s3:GetObject"`, `"ec2:StartInstances"` |
| **Resource** | Which specific AWS resources | `"arn:aws:s3:::my-bucket/*"` |
| **Condition** | Optional extra requirements | Source IP, time of day, MFA required |

### The Deny-Always-Wins Rule

When AWS evaluates policies, it follows a simple order:

1. By default, everything is **denied** (implicit deny)
2. An explicit **Allow** overrides the implicit deny
3. An explicit **Deny** overrides any Allow

**If any policy says Deny, the answer is Deny. Period.**

| Type | Created By | Reusable? | Best For |
|------|-----------|-----------|----------|
| **AWS Managed** | AWS | Yes | Common use cases (ReadOnlyAccess, S3FullAccess) |
| **Customer Managed** | You | Yes | Custom permissions specific to your organization |
| **Inline** | You | No (embedded in one entity) | Rare, strict 1-to-1 relationships |

**Best practice:** Start with AWS managed policies. Create customer managed policies when you need finer control. Avoid inline policies unless you have a specific reason.

## IAM Roles

Roles provide **temporary credentials** instead of permanent ones. They are used when:

| Scenario | Example |
|----------|---------|
| **AWS service to service** | An EC2 instance needs to read files from S3 |
| **Cross-account access** | A user in Account A needs to access resources in Account B |
| **Federated users** | Employees sign in through your company's Active Directory |
| **Temporary elevation** | A developer needs temporary admin access for a deployment |

When an entity **assumes** a role, AWS gives it temporary security credentials that **expire automatically** (usually within 1-12 hours). This is much safer than storing permanent access keys.

### Roles vs. Users: When to Use Each

| Use an IAM User When... | Use an IAM Role When... |
|--------------------------|------------------------|
| A human needs long-term AWS console access | An AWS service needs to access another service |
| An application needs permanent API keys (rare) | Someone needs temporary, time-limited access |
| You need to track individual actions in CloudTrail | External users or accounts need access to your resources |

**When in doubt, use a role.** Temporary credentials are always more secure than permanent ones.

## Key Takeaways

- Create **one IAM user per person** and manage permissions through **groups**, not individual users
- IAM policies are JSON documents with **Effect**, **Action**, and **Resource** as core elements
- **Deny always wins** -- an explicit Deny overrides any Allow
- Three policy types: **AWS Managed** (pre-built), **Customer Managed** (your custom policies), and **Inline** (embedded, avoid when possible)
- Use **IAM Roles** for service-to-service access, cross-account access, and temporary credentials
- Groups cannot be nested, but a user can belong to multiple groups
