# AWS Identity & Access Management (IAM) Basics

**AWS IAM** is the service that controls **who** can access your AWS account and **what** they can do. Every API call to AWS goes through IAM for permission checks. If you skip IAM, you leave the front door to your cloud wide open.

Think of IAM as the security badge system in an office building. Each person gets a badge (identity) that opens only the doors they need (permissions). The security guard (IAM) checks every badge at every door.

## The Root Account Problem

When you create an AWS account, you get a **root account** tied to your email address. This account has **unlimited power** -- it can do anything, including closing the entire account and deleting all data.

**Only use the root account for tasks that require it**, such as changing your AWS support plan, closing the account, or restoring IAM permissions. For everything else, create IAM users.

## The Principle of Least Privilege

The most important security rule in IAM: **give users only the permissions they need to do their job -- nothing more.**

If a developer needs to upload files to S3, give them S3 write access. Do not give them the ability to delete databases, modify billing, or manage other users. The less access someone has, the less damage can happen if their credentials are stolen.

| Building Block | What It Is | Example |
|----------------|-----------|---------|
| **User** | An identity for a person or application | "developer-jane" who writes code |
| **Group** | A collection of users with shared permissions | "Developers" group with S3 and Lambda access |
| **Policy** | A JSON document that defines what actions are allowed or denied | "Allow read access to S3 bucket X" |
| **Role** | Temporary credentials for users, services, or external accounts | EC2 instance that needs to read from S3 |

### Users

An IAM user is an identity with permanent credentials (password and/or access keys). Each person who needs AWS access should get their own IAM user. Never share credentials between team members.

### Groups

Groups let you manage permissions for many users at once. Instead of attaching policies to each user individually, attach them to a group and add users to it. When a user changes teams, just move them to a different group.

### Policies

Policies are JSON documents that define permissions. They specify which **actions** are allowed on which **resources** under what **conditions**. We will dive deeper into policies in the next lesson.

### Roles

Roles provide **temporary** security credentials. They are used when:

- An AWS service needs to access another service (EC2 reading from S3)
- Users from another AWS account need access to your resources
- Federated users (from your company's identity system) need AWS access

Roles do not have passwords or permanent access keys. They issue short-lived tokens that expire automatically.

## IAM Is Global and Free

Two important facts about IAM:

- **Global** -- IAM users, groups, roles, and policies work across all AWS Regions. You create them once and they apply everywhere.
- **Free** -- IAM costs nothing. You never pay for creating users, groups, roles, or policies. AWS considers access management a core security feature, not an add-on.

## Key Takeaways

- IAM controls who can access your AWS account and what actions they can perform
- The **root account** has unlimited power -- secure it with MFA and never use it for daily tasks
- Follow the **principle of least privilege**: give only the permissions needed, nothing more
- IAM has four building blocks: **Users** (identities), **Groups** (collections of users), **Policies** (permission rules), and **Roles** (temporary access)
- IAM is **global** (not Region-specific) and completely **free** to use
