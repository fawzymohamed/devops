# Security & Compliance Foundations - Quick Reference

## Shared Responsibility Model

| Responsibility | AWS (OF the Cloud) | Customer (IN the Cloud) |
|---------------|-------------------|------------------------|
| **Physical** | Data centers, hardware, networking | N/A |
| **Infrastructure** | Hypervisor, host OS, managed service internals | OS patches (EC2), runtime config |
| **Data** | N/A | All customer data, encryption settings |
| **Access** | N/A | IAM users, roles, policies, security groups |
| **Application** | N/A | Code, libraries, dependencies |

**Memory trick:** If you can configure it in the AWS console, it is probably your responsibility.

### Responsibility by Service Type

| Layer | EC2 (IaaS) | RDS (PaaS) | Lambda (Serverless) | S3 (Managed) |
|-------|-----------|-----------|-------------------|-------------|
| Your data | You | You | You | You |
| IAM & access | You | You | You | You |
| OS patching | You | AWS | AWS | AWS |
| Runtime | You | AWS | AWS | AWS |
| Infrastructure | AWS | AWS | AWS | AWS |

> **Rule:** The more managed the service, the less you manage.

## IAM at a Glance

### Core Building Blocks

| Entity | What It Is | Credentials | Use For |
|--------|-----------|-------------|---------|
| **User** | Identity for a person or app | Password + access keys (permanent) | Individual human access |
| **Group** | Collection of users | N/A (inherits from policies) | Organizing users by job function |
| **Role** | Temporary identity | Short-lived tokens (auto-expire) | Service-to-service, cross-account, federation |
| **Policy** | JSON permission document | N/A | Defining allowed/denied actions |

### IAM Policy Structure

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow | Deny",
    "Action": ["service:APICall"],
    "Resource": "arn:aws:service:::resource",
    "Condition": { }
  }]
}
```

### Policy Evaluation Order

1. **Default:** Everything denied (implicit deny)
2. **Allow** overrides implicit deny
3. **Explicit Deny** overrides any Allow -- **Deny always wins**

### Three Policy Types

| Type | Created By | Reusable? | Best For |
|------|-----------|-----------|----------|
| **AWS Managed** | AWS | Yes | Common use cases (ReadOnlyAccess) |
| **Customer Managed** | You | Yes | Custom org-specific permissions |
| **Inline** | You | No (embedded) | Strict 1-to-1 relationships (avoid) |

### IAM Key Rules

- One user per person -- never share accounts
- Groups **cannot** be nested, but users can belong to multiple groups
- Use **roles** (not access keys) for service-to-service access
- IAM is **global** (works across all Regions) and **free**

## Multi-Factor Authentication (MFA)

**MFA = Something you know (password) + Something you have (device)**

### MFA Device Types

| Type | Example | Cost | Phishing-Resistant? |
|------|---------|------|-------------------|
| **Virtual MFA** | Google Authenticator, Authy | Free | No |
| **Hardware TOTP** | Gemalto key fob | ~$13-20 | No |
| **FIDO2 Security Key** | YubiKey, Titan Key | ~$25-50 | **Yes** |

### MFA Rules

- **Root account MFA is non-negotiable** -- enable it first
- Enable MFA on every IAM user, not just root
- Use a dedicated MFA device for root (separate from personal phone)
- Store backup codes in a secure location

## AWS Organizations & Consolidated Billing

### Organization Hierarchy

```
Root
├── Management Account (pays the bill, immune to SCPs)
├── Security OU
│   └── Member Accounts
├── Production OU
│   └── Member Accounts
└── Development OU
    └── Member Accounts
```

### SCPs vs. IAM Policies

| Feature | SCPs | IAM Policies |
|---------|------|-------------|
| **Scope** | Entire accounts / OUs | Users, groups, roles |
| **Can grant?** | No -- restrict only | Yes |
| **Apply to management account?** | No | Yes |
| **Purpose** | Set maximum allowed actions | Define actual permissions |

> **Both** the SCP and IAM policy must allow an action for it to succeed.

### Consolidated Billing Benefits

- **Single bill** for all accounts
- **Volume discounts** -- combined usage hits lower pricing tiers
- **RI / Savings Plan sharing** across accounts
- **Cost allocation tags** for tracking spend by project or team

## Compliance Programs

| Program | Covers | Required When |
|---------|--------|--------------|
| **SOC 1** | Financial reporting controls | Cloud use affects financial statements |
| **SOC 2** | Security, availability, confidentiality | Most common -- "prove you are secure" |
| **SOC 3** | Public summary of SOC 2 | Marketing / public trust |
| **ISO 27001** | Info security management (global) | International security certification needed |
| **PCI DSS** | Credit card data protection | You store, process, or transmit card data |
| **HIPAA** | Patient health information (US) | Healthcare data -- requires signed BAA |
| **FedRAMP** | US federal government security | Government agencies or contractors |
| **GDPR** | European personal data | Handle EU citizen data -- requires DPA |

> **AWS compliance does not make you compliant.** You must still secure your own data and configurations.

## Encryption Quick Reference

### At Rest vs. In Transit

| Aspect | At Rest | In Transit |
|--------|---------|-----------|
| **Protects** | Stored data (disks, S3, DBs) | Data moving over the network |
| **Uses** | AES-256 encryption | SSL/TLS protocols (HTTPS) |
| **Managed by** | AWS KMS | AWS Certificate Manager |

### KMS Key Types

| Key Type | Managed By | Rotation | Cost |
|----------|-----------|----------|------|
| **AWS owned** | AWS (hidden) | AWS handles | Free |
| **AWS managed** | AWS (visible) | Auto yearly | Free + API costs |
| **Customer managed** | You | You configure | $1/month + API |

### S3 Encryption Options

| Option | Key Management | Cost | Notes |
|--------|---------------|------|-------|
| **SSE-S3** | S3 manages everything | Free | Default on all new buckets |
| **SSE-KMS** | KMS manages, you control policy | KMS API costs | Audit trail, key rotation |
| **SSE-C** | You provide key each request | Free | Full control, strict compliance |
| **Client-side** | You encrypt before upload | Free | AWS never sees plaintext |

### Server-Side vs. Client-Side

- **Server-side:** AWS encrypts after data arrives (SSE-S3, SSE-KMS, SSE-C)
- **Client-side:** You encrypt before sending -- maximum control

## AWS Artifact

| Section | Contains | Action |
|---------|----------|--------|
| **Reports** | SOC 1/2/3, ISO 27001, PCI DSS, FedRAMP | Download (some require NDA) |
| **Agreements** | BAA (HIPAA), DPA (GDPR) | Review and accept |

- **Free** and available in every AWS account
- Organization-level agreements apply to **all member accounts**
- SOC 2 and SOC 1 reports require NDA; SOC 3 is public

## Interview Quick Hits

| Question | Answer |
|----------|--------|
| What is the Shared Responsibility Model? | AWS secures the cloud infrastructure (OF the cloud); you secure your data, access, and configurations (IN the cloud) |
| What is the principle of least privilege? | Give users only the minimum permissions needed for their job -- nothing more |
| When should you use an IAM role vs. user? | Roles for temporary access (service-to-service, cross-account, federation). Users for human long-term console access |
| What happens when an IAM policy has both Allow and Deny? | Explicit Deny always wins over Allow |
| What are SCPs and can they grant permissions? | Service Control Policies set maximum allowed actions for accounts. They restrict only -- they cannot grant permissions |
| What makes FIDO2 security keys special? | They are phishing-resistant because they cryptographically verify the website, not just generate a code |
| What is the difference between SSE-S3 and SSE-KMS? | SSE-S3 is free and fully managed by S3. SSE-KMS gives you key policy control, audit trails via CloudTrail, and rotation management |
| What is AWS Artifact used for? | A free portal to download compliance reports (SOC, ISO, PCI) and accept agreements (BAA for HIPAA, DPA for GDPR) |
