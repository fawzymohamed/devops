# AWS Organizations & Consolidated Billing

As companies grow, they do not put everything in a single AWS account. They create **multiple accounts** -- one for development, one for production, one for security logging, and more. **AWS Organizations** lets you manage all these accounts from a single place with centralized billing and security controls.

Think of it like a company with multiple departments. Each department has its own budget and office space (separate accounts), but the CEO (management account) sees the full picture and sets company-wide rules.

## Why Multiple Accounts?

| Reason | Example |
|--------|---------|
| **Environment isolation** | Separate dev, staging, and production so a mistake in dev cannot break production |
| **Security boundaries** | A compromised dev account cannot access production data |
| **Billing separation** | Track costs per team, project, or department |
| **Compliance** | Keep regulated workloads (HIPAA, PCI) in a dedicated account with strict controls |
| **Service limits** | Each account has its own service quotas, avoiding conflicts between teams |

## How AWS Organizations Works

An organization has a **management account** (the one that creates the organization) and one or more **member accounts**. The management account is the central hub for billing, account creation, and policy management.

## Organizational Units (OUs)

**OUs** are containers that group AWS accounts. You can nest OUs inside other OUs to create a hierarchy that matches your company structure.

**Example structure:**

```
Root
├── Security OU
│   ├── Log Archive Account
│   └── Security Tooling Account
├── Production OU
│   ├── App Team A Account
│   └── App Team B Account
└── Development OU
    ├── Dev Sandbox Account
    └── Testing Account
```

Policies attached to an OU apply to **every account inside it**. If you attach a policy to the Production OU that blocks deleting S3 buckets, no account in Production can delete S3 buckets.

## Service Control Policies (SCPs)

SCPs are **guardrails** that set the **maximum permissions** available to accounts. They do not grant access -- they only restrict it.

**How SCPs and IAM work together:**

For a user to perform an action, the action must be allowed by **both** the SCP on the account and the IAM policy on the user. If either one denies it, the action is blocked.

**Common SCP examples:**

- Block all access outside specific AWS Regions (data sovereignty)
- Prevent member accounts from leaving the organization
- Block the ability to disable CloudTrail (audit logging)
- Restrict which EC2 instance types can be launched

**Important:** SCPs **never apply to the management account**. This is why you should not run workloads in the management account -- use it only for organization administration.

## Consolidated Billing

All accounts in an organization get a **single combined bill**. The management account pays for everything and sees usage across all accounts.

### Benefits of Consolidated Billing

| Benefit | How It Works |
|---------|-------------|
| **Single bill** | One payment method, one invoice for all accounts |
| **Volume discounts** | Combined usage across accounts reaches higher pricing tiers faster |
| **RI and Savings Plan sharing** | Reserved Instances bought in one account can apply to matching usage in other accounts |
| **Cost visibility** | See per-account spending breakdowns in one place |

**Example of volume discounts:** If three accounts each use 50 TB of S3 storage, you pay as if you used 150 TB total. The first 50 TB costs more per GB than the next 100 TB, so the combined bill is cheaper than three separate bills.

### Cost Allocation Tags

You can tag resources across accounts (like `project:web-app` or `team:marketing`) and then view costs grouped by those tags. This helps finance teams track spending by project, department, or environment.

## Key Takeaways

- AWS Organizations manages **multiple AWS accounts** from a central management account
- **Organizational Units (OUs)** group accounts and can be nested for hierarchical control
- **Service Control Policies (SCPs)** set guardrails on maximum permissions -- they restrict but never grant access
- Both the SCP and IAM policy must allow an action for it to succeed
- SCPs do not apply to the management account -- do not run workloads there
- **Consolidated billing** combines all account charges, shares volume discounts, and allows Reserved Instance sharing across accounts
