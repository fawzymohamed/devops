# Migration Strategies (6 Rs)

When moving workloads to AWS, not every application should be migrated the same way. AWS defines **six common migration strategies** -- called the **6 Rs** -- that give you a framework for deciding how to handle each workload.

## The 6 Rs at a Glance

| Strategy | Nickname | Effort | Cloud Benefit |
|----------|----------|--------|---------------|
| **Rehost** | Lift and shift | Low | Low |
| **Replatform** | Lift, tinker, and shift | Low-Medium | Medium |
| **Repurchase** | Drop and shop | Medium | Medium |
| **Refactor** | Re-architect | High | High |
| **Retire** | Shut it down | None | Cost savings |
| **Retain** | Keep as-is | None | None (yet) |

## Each Strategy Explained

### Rehost (Lift and Shift)

Move your application to AWS **without making changes**. You take your existing servers, databases, and configurations and run them on EC2 instances in the cloud.

- **Best for:** Large-scale migrations where speed matters more than optimization
- **Example:** Move an on-premises web server to an EC2 instance with the same OS and configuration
- **Tools:** AWS Application Migration Service

Rehost is the fastest path to the cloud. You can optimize later once the workload is running on AWS.

### Replatform (Lift, Tinker, and Shift)

Move to the cloud with **small optimizations** that take advantage of AWS managed services, without changing the core application architecture.

- **Best for:** Applications that benefit from managed services with minimal rework
- **Example:** Migrate a self-managed MySQL database to Amazon RDS instead of running MySQL on EC2
- **Benefit:** Less operational overhead while keeping the same application code

### Repurchase (Drop and Shop)

**Replace** the existing application with a different product, usually a SaaS solution.

- **Best for:** Legacy applications where a modern SaaS alternative exists
- **Example:** Replace a self-hosted CRM with Salesforce, or switch from on-premises email to Microsoft 365
- **Tradeoff:** May need user training and data migration to the new platform

### Refactor (Re-architect)

**Redesign** the application from the ground up to use cloud-native features like serverless, containers, or microservices.

- **Best for:** Applications that need better scalability, performance, or agility
- **Example:** Break a monolithic app into microservices running on ECS, with DynamoDB replacing a relational database and Lambda handling event processing
- **Tradeoff:** Takes the most time and effort but delivers the most long-term value

### Retire

**Shut down** applications you no longer need. During migration planning, most organizations discover 10-20% of their portfolio is no longer useful.

- **Best for:** Redundant, outdated, or unused applications
- **Example:** Decommission an old reporting tool that only three people used and has been replaced
- **Benefit:** Reduces costs, security risks, and maintenance burden

### Retain

**Keep** the application where it is for now. Some workloads are not ready to migrate due to compliance requirements, recent upgrades, or unresolved dependencies.

- **Best for:** Applications with regulatory constraints, recent hardware investments, or complex dependencies
- **Example:** A mainframe application that requires a multi-year modernization plan stays on-premises until the plan is ready
- **Note:** Retain does not mean "never migrate" -- it means "not right now"

## Choosing the Right Strategy

| Scenario | Recommended Strategy |
|----------|---------------------|
| Need to migrate fast with minimal risk | Rehost |
| Want quick wins from managed services | Replatform |
| A good SaaS replacement exists | Repurchase |
| Application needs major performance or scalability improvements | Refactor |
| No one uses the application anymore | Retire |
| Compliance or dependencies prevent migration | Retain |

**Practical tip:** Most organizations use a **mix of strategies**. Start by identifying what to retire and retain. Then rehost the bulk of workloads for speed. Replatform and refactor high-value applications over time.

## Key Takeaways

- The 6 Rs give you a framework for deciding how to migrate each workload
- **Rehost** is fastest (lift and shift); **Refactor** delivers the most long-term value (re-architect)
- **Retire** unused applications first to save money before migrating anything
- **Retain** is a valid choice -- not every workload needs to move right now
- Most migrations use a mix of strategies across different applications
