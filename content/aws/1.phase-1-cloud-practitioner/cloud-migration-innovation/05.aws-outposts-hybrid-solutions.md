# AWS Outposts & Hybrid Solutions

**AWS Outposts** brings AWS infrastructure and services to your own data center or edge location. AWS delivers the hardware, installs it, and manages it -- you use the same AWS APIs, tools, and services you would in the cloud, but the servers sit in your building.

## Why Hybrid Cloud?

Not every workload can run in the cloud. Some common reasons to keep workloads on-premises:

- **Data residency** -- Laws or regulations require data to stay in a specific physical location
- **Low latency** -- Factory equipment or trading systems need single-digit millisecond response times to local systems
- **Local processing** -- Process data locally first, then send results to the cloud

Outposts lets you run AWS services in these situations without giving up cloud benefits like managed services, consistent APIs, and unified management.

## Outposts Form Factors

### Outposts Racks

A full **42U rack** of AWS compute and storage hardware, delivered and installed in your data center.

| Feature | Details |
|---------|---------|
| **Size** | Standard 42U industry rack |
| **Scale** | 1 to 96 racks |
| **Power** | 10, 15, or 30 kVA options |
| **Services** | EC2, ECS, EKS, EBS, S3, RDS, ALB, Route 53, and more |
| **Best for** | Data centers with space and power for full racks |

### Outposts Servers

A **1U or 2U server** that fits into your existing rack -- ideal for locations with limited space.

| Feature | Details |
|---------|---------|
| **Size** | 1U (Graviton2 processor) or 2U (Intel Xeon) |
| **Power** | 1-2 kVA |
| **Services** | EC2, ECS, and AWS IoT Greengrass |
| **Best for** | Retail stores, branch offices, factories, and edge locations |

## How Outposts Works

Every Outposts deployment connects back to a **parent AWS Region**. This connection is used for:

- **Management** -- AWS monitors hardware health and pushes updates remotely
- **Extended services** -- Use Regional services like IAM, CloudWatch, and CloudFormation to manage Outposts resources
- **Data sync** -- Replicate data between Outposts and the Region as needed

You manage Outposts resources using the **same AWS Console, CLI, and SDKs** you use for cloud resources. An EC2 instance on Outposts works the same as one in the cloud -- same APIs, same tools, same experience.

## Common Use Cases

### Manufacturing

Run machine learning models on the factory floor to detect defects in real time. The data stays local for low-latency processing, and results sync to the cloud for long-term analysis.

### Healthcare

Keep patient data on-premises to meet HIPAA and data residency requirements while using AWS services like EC2 and RDS for application hosting.

### Retail

Deploy Outposts Servers in stores for local inventory management and point-of-sale systems that work even when internet connectivity is unreliable.

### Financial Services

Run latency-sensitive trading applications on Outposts for fast local processing while using the parent Region for analytics, reporting, and backup.

## Other AWS Hybrid Services

Outposts is not the only hybrid option. AWS offers several services for connecting on-premises and cloud:

| Service | Purpose |
|---------|---------|
| **AWS Outposts** | Run AWS services on-premises with managed hardware |
| **AWS Local Zones** | Extend AWS Regions to metro areas for low-latency access |
| **AWS Wavelength** | Deploy applications at telecom carrier 5G edge locations |
| **AWS Snow Family** | Move data physically and run compute in disconnected environments |

## Key Takeaways

- AWS Outposts puts **AWS hardware in your data center**, managed by AWS
- Two form factors: **Racks** (full 42U rack for data centers) and **Servers** (1U/2U for edge locations)
- Use Outposts when you need **data residency**, **low latency**, or **local processing**
- Outposts connects to a parent AWS Region for management, updates, and access to additional services
- You use the **same AWS APIs and tools** whether resources are in the cloud or on Outposts
