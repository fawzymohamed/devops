# Additional AWS Services - Quick Reference

## Messaging Services at a Glance

| Service | Model | Delivery | Best For |
|---------|-------|----------|----------|
| **SQS** | Queue (point-to-point) | Consumer polls queue | Decoupling two components, buffering |
| **SNS** | Pub/Sub (fan-out) | Push to all subscribers | One event to many recipients |
| **EventBridge** | Event bus (rules-based) | Route by content rules | Multi-source event routing, SaaS integration |

### SQS Queue Types

| Feature | Standard Queue | FIFO Queue |
|---------|---------------|------------|
| **Ordering** | Best-effort | Exact order guaranteed |
| **Delivery** | At least once (possible duplicates) | Exactly once |
| **Throughput** | Nearly unlimited | Up to 3,000 msg/sec (batched) |
| **Use case** | Log processing, high throughput | Financial transactions, order processing |

### SQS Key Features

- **Visibility timeout** -- hides message while consumer processes it (default 30s)
- **Dead-letter queue** -- captures messages that fail after multiple retries
- **Retention** -- up to 14 days (default 4 days)

### SNS Subscriber Types

Email, SMS, Lambda, SQS, HTTP/HTTPS

**Fan-out pattern:** Publish to SNS topic, which delivers to multiple SQS queues for parallel processing.

### EventBridge Extras

- **Schema registry** -- auto-discovers event formats
- **Content-based filtering** -- route on fields inside the event payload
- **SaaS integration** -- Shopify, Zendesk, Datadog
- **Scheduling** -- cron-style scheduling without extra infrastructure

## AWS Step Functions

### Workflow Types

| Feature | Standard | Express |
|---------|----------|---------|
| **Max duration** | 1 year | 5 minutes |
| **Execution rate** | 2,000/sec | 100,000/sec |
| **Delivery** | Exactly once | At least once |
| **Best for** | Order processing, approvals, ETL | IoT data, streaming transforms |

### State Types

| State | Purpose |
|-------|---------|
| **Task** | Call an AWS service or API |
| **Choice** | Branching logic (if/else) |
| **Parallel** | Run branches at the same time |
| **Wait** | Pause for a duration or timestamp |
| **Map** | Iterate over a list |
| **Pass** | Transform data between steps |
| **Succeed / Fail** | End the workflow |

### Error Handling

- **Retry** -- re-run failed state with configurable intervals
- **Catch** -- redirect to a fallback state when retries exhaust

### Other Integration Services

| Service | Purpose |
|---------|---------|
| **Amazon MQ** | Managed message broker (ActiveMQ, RabbitMQ) -- MQTT, AMQP, STOMP |
| **Amazon AppFlow** | Automated data flow between SaaS apps and AWS |
| **AWS AppSync** | Managed GraphQL and Pub/Sub API service |

## Container Services

### ECS vs. EKS

| Factor | ECS | EKS |
|--------|-----|-----|
| **Orchestration** | AWS-native | Kubernetes (open-source) |
| **Learning curve** | Lower | Higher |
| **Portability** | AWS only | Multi-cloud, on-premises |
| **Ecosystem** | AWS tools | Large Kubernetes ecosystem |
| **Best for** | Teams new to containers | Teams with Kubernetes experience |
| **Fargate support** | Yes | Yes |

### Launch Types

| Type | Who Manages Servers | Control | Best For |
|------|-------------------|---------|----------|
| **Fargate** | AWS (serverless) | Less | Beginners, no server management |
| **EC2** | You | Full | GPU access, specific instance types |

### Key ECS Concepts

| Concept | What It Is |
|---------|-----------|
| **Cluster** | Group of compute resources |
| **Task definition** | Blueprint: image, CPU, memory, ports |
| **Task** | Running instance of a task definition |
| **Service** | Keeps desired task count running |

**Amazon ECR** -- fully managed Docker container registry with IAM integration.

### Containers vs. Lambda

| Factor | Containers (ECS/EKS) | Lambda |
|--------|----------------------|--------|
| **Execution time** | No limit | Max 15 minutes |
| **Control** | Full (runtime, OS) | AWS manages everything |
| **Best for** | Long-running apps, microservices | Short event-driven tasks |

## AWS CloudFormation (IaC)

### Core Concepts

| Concept | What It Is |
|---------|-----------|
| **Template** | YAML or JSON file describing resources |
| **Stack** | Collection of resources created from a template |
| **Change set** | Preview of proposed changes before applying |

### Template Sections

| Section | Required? | Purpose |
|---------|-----------|---------|
| **Resources** | Yes | AWS resources to create |
| **Parameters** | No | Input values at stack creation |
| **Outputs** | No | Values to display after creation |
| **Mappings** | No | Static lookup tables (e.g., AMI per Region) |
| **Description** | No | What the template does |

### Key Behaviors

- **Delete stack** -- all resources in the stack are deleted together
- **Failed create** -- automatic rollback to previous state
- **Dependency ordering** -- CloudFormation figures out creation order
- **Free** -- you pay only for the resources it creates

## CloudWatch vs. CloudTrail

| Aspect | CloudWatch | CloudTrail |
|--------|-----------|------------|
| **Tracks** | Performance metrics (how things run) | API calls (who did what) |
| **Focus** | Monitoring and alerting | Auditing and security |
| **Example** | "Is CPU above 80%?" | "Who deleted the database?" |
| **Output** | Metrics, alarms, dashboards, logs | Event logs (JSON) |

### CloudWatch Key Concepts

| Concept | What It Is |
|---------|-----------|
| **Metric** | Time-series data points (e.g., CPUUtilization) |
| **Alarm** | Watches a metric, triggers actions on threshold breach |
| **Dashboard** | Visual display of metrics from multiple services |
| **Logs** | Collect, store, and search log data |
| **Custom metrics** | Your own app metrics sent via SDK or agent |

- **Basic monitoring:** every 5 min (free)
- **Detailed monitoring:** every 1 min (paid)

**Alarm actions:** SNS notification, Auto Scaling, EC2 stop/terminate/reboot

### CloudTrail Key Concepts

| Concept | What It Is |
|---------|-----------|
| **Event** | Record of an API call (who, what, when, where) |
| **Event History** | Last 90 days of management events (free, no setup) |
| **Trail** | Delivers logs to S3 / CloudWatch Logs for long-term storage |
| **Insights** | Detects unusual API activity patterns |

**Event types:**
- **Management events** -- control plane (create/delete resources) -- free, logged by default
- **Data events** -- data plane (S3 object reads/writes, Lambda invocations) -- extra charge, disabled by default

## AWS Systems Manager

### Key Capabilities

| Capability | Purpose |
|------------|---------|
| **Run Command** | Execute commands on many instances remotely (no SSH) |
| **Session Manager** | Secure browser/CLI shell -- no open ports, no SSH keys |
| **Parameter Store** | Store config values and encrypted secrets |
| **Patch Manager** | Automate OS/software patching with compliance reports |
| **Automation** | Run predefined runbooks (AMI creation, remediation) |

### SSM Agent Requirements

- SSM Agent installed on the instance (pre-installed on most Amazon AMIs)
- IAM role with Systems Manager permissions attached
- Outbound internet access to reach the Systems Manager API

### Parameter Store Types

| Type | Example | Encryption |
|------|---------|------------|
| **String** | Feature flag: `enableNewUI = true` | Optional |
| **StringList** | Allowed regions: `us-east-1, eu-west-1` | Optional |
| **SecureString** | Database password | KMS encrypted |

**Parameter Store vs. Secrets Manager:** Parameter Store is free (standard) for general config. Secrets Manager costs more but adds automatic secret rotation.

### Session Manager vs. SSH

| Feature | SSH | Session Manager |
|---------|-----|----------------|
| **Open ports** | Port 22 required | No inbound ports |
| **Key management** | SSH keys | IAM policies |
| **Audit trail** | Manual setup | CloudTrail + optional S3 logging |

## Amazon DynamoDB

### SQL vs. DynamoDB

| Feature | Relational (SQL) | DynamoDB (NoSQL) |
|---------|-----------------|------------------|
| **Schema** | Fixed columns | Flexible, schema-less |
| **Joins** | Supported | Not supported |
| **Scaling** | Vertical (bigger server) | Horizontal (automatic) |
| **Best for** | Complex queries with relationships | High-speed reads/writes at scale |

### Data Model

| Concept | SQL Equivalent |
|---------|---------------|
| **Table** | Table |
| **Item** | Row |
| **Attribute** | Column |

### Primary Key Options

- **Simple** -- partition key only (e.g., `userId`)
- **Composite** -- partition key + sort key (e.g., `userId` + `orderDate`)

### Capacity Modes

| Mode | How It Works | Best For |
|------|-------------|----------|
| **On-Demand** | Auto-scales, pay per request | Unpredictable workloads |
| **Provisioned** | Set RCU/WCU in advance | Steady workloads (cheaper) |

Switch between modes once every 24 hours.

### Key Features

| Feature | What It Does |
|---------|-------------|
| **Global Tables** | Multi-Region replication for low-latency global access |
| **DynamoDB Streams** | Time-ordered log of changes; trigger Lambda on inserts/updates/deletes |
| **DAX** | In-memory cache -- reduces reads from milliseconds to microseconds |
| **PITR** | Restore table to any second in the last 35 days |

- Replicates across **3 AZs** automatically (99.99% availability)
- **Free tier:** 25 GB storage, 25 RCU, 25 WCU (permanent, not 12-month)

## Interview Quick Hits

| Question | Answer |
|----------|--------|
| SQS vs. SNS? | SQS is a queue for one consumer (pull-based). SNS is pub/sub for many subscribers (push-based). Combine them for fan-out |
| When would you use EventBridge over SNS? | When you need content-based event filtering, SaaS integration, or routing events from many different sources by rules |
| Standard vs. Express Step Functions? | Standard runs up to 1 year with exactly-once delivery. Express runs up to 5 minutes with higher throughput for short tasks |
| ECS vs. EKS? | ECS is simpler and AWS-native. EKS runs Kubernetes for teams with K8s experience or multi-cloud needs |
| What is a CloudFormation stack? | A collection of AWS resources managed as a single unit -- create, update, or delete them all together from a template |
| CloudWatch vs. CloudTrail? | CloudWatch monitors performance (metrics, alarms). CloudTrail audits API calls (who did what, when) |
| What is Parameter Store? | A Systems Manager feature that stores config values and encrypted secrets for applications to retrieve at runtime |
| When to use DynamoDB vs. RDS? | DynamoDB for high-speed key-value access at scale. RDS for complex SQL queries with joins and relationships |

## Quick Decision Guide

**Choose SQS when:** You need to buffer messages between two services and process them one at a time.

**Choose SNS when:** One event needs to reach multiple subscribers at once (email, SMS, Lambda, SQS).

**Choose EventBridge when:** You route events from many sources using content-based rules.

**Choose Step Functions when:** You coordinate multi-step workflows with branching, retries, and error handling.

**Choose ECS when:** You need containers on AWS with a simple learning curve.

**Choose EKS when:** Your team already knows Kubernetes or you need multi-cloud portability.

**Choose CloudFormation when:** You want to define and version your infrastructure as code.

**Choose DynamoDB when:** You need fast, serverless NoSQL with simple access patterns at any scale.
