# AWS Systems Manager Overview

**AWS Systems Manager** gives you a single console to manage your EC2 instances and on-premises servers at scale. Instead of connecting to each server individually to run commands, install patches, or check configurations, Systems Manager lets you do it all from one place.

## The Problem It Solves

Managing servers gets harder as you scale. With dozens or hundreds of instances, you face challenges like:

- Running the same command on 50 servers at once
- Keeping all servers patched with the latest security updates
- Connecting to instances securely without exposing SSH ports
- Storing database passwords and API keys safely

Systems Manager bundles solutions to all these problems into a single service.

## How It Works: The SSM Agent

Systems Manager communicates with your instances through the **SSM Agent** -- a lightweight software component that runs on each managed server.

| Requirement | Detail |
|-------------|--------|
| **SSM Agent installed** | Must be present on the instance |
| **IAM role attached** | The instance needs an IAM role with Systems Manager permissions |
| **Outbound internet** | The agent needs to reach the Systems Manager API endpoint |

The SSM Agent comes **pre-installed** on most Amazon Linux, Ubuntu, and Windows AMIs. For on-premises servers, you install it manually.

Once the agent is running, your instance appears as a **managed node** in the Systems Manager console.

## Key Capabilities

### Run Command

**Run Command** lets you execute commands on one or many instances remotely -- without SSH or Remote Desktop.

**Example use cases:**

- Install or update software on 100 instances at once
- Collect diagnostic information from all servers
- Restart a service across your fleet

You select target instances (by tag, instance ID, or all), choose a command document, and Systems Manager runs it. Results appear in the console.

### Session Manager

**Session Manager** gives you a secure shell to your instances through the browser or CLI.

**Why it is better than SSH:**

| SSH | Session Manager |
|-----|----------------|
| Requires port 22 open in security group | No inbound ports needed |
| Requires SSH key management | Uses IAM for access control |
| No audit trail by default | All sessions are logged to CloudTrail and optionally S3 |

Session Manager improves security because you do not need bastion hosts, open ports, or SSH keys.

### Parameter Store

**Parameter Store** is a secure, centralized place to store configuration data and secrets.

**What you can store:**

| Type | Example | Encryption |
|------|---------|------------|
| **String** | Feature flag: `enableNewUI = true` | Optional |
| **StringList** | Allowed regions: `us-east-1, eu-west-1` | Optional |
| **SecureString** | Database password: `P@ssw0rd!` | Encrypted with KMS |

Your applications retrieve values from Parameter Store at runtime using the AWS SDK. This keeps secrets out of your code and makes it easy to change settings without redeployment.

**Parameter Store vs. Secrets Manager:** Parameter Store is free for standard parameters and handles general configuration. AWS Secrets Manager costs more but adds automatic rotation for database credentials. Use Parameter Store for most configuration; use Secrets Manager when you need automatic secret rotation.

### Patch Manager

**Patch Manager** automates the process of patching your managed instances with security and software updates.

**How it works:**

1. Define a **patch baseline** -- which patches to approve (critical, security, all)
2. Create a **maintenance window** -- when patching should happen (e.g., Sunday 2:00 AM)
3. Patch Manager applies approved patches to target instances during the window
4. Review **compliance reports** to see which instances are patched and which are not

### Automation

**Automation** runs predefined runbooks to perform common tasks like:

- Creating a golden AMI (image) from a running instance
- Remediating non-compliant resources automatically
- Starting or stopping instances on a schedule

Runbooks are step-by-step documents that Systems Manager follows. AWS provides many built-in runbooks, and you can create your own.

## Key Takeaways

- Systems Manager provides one console to manage EC2 instances and on-premises servers at scale
- The **SSM Agent** must be installed and an IAM role attached for an instance to be managed
- **Run Command** executes commands remotely on many instances without SSH
- **Session Manager** provides secure browser-based access without open ports or SSH keys
- **Parameter Store** stores configuration values and encrypted secrets for your applications
- **Patch Manager** automates OS and software patching with compliance reporting
- **Automation** runs runbooks for common operational tasks like AMI creation and remediation
