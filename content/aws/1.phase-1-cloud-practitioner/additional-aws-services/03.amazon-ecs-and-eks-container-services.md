# Amazon ECS & EKS Container Services

**Containers** package your application code, libraries, and runtime into a single portable unit that runs the same way everywhere. AWS provides two managed services to run containers at scale: **Amazon ECS** (Elastic Container Service) and **Amazon EKS** (Elastic Kubernetes Service).

## What Are Containers?

A container is a lightweight, standalone package that includes everything your application needs to run: code, runtime, system tools, and libraries. Unlike virtual machines, containers share the host operating system kernel, making them faster to start and more efficient with resources.

| Virtual Machine | Container |
|-----------------|-----------|
| Includes a full guest OS | Shares the host OS kernel |
| Boots in minutes | Starts in seconds |
| Uses GBs of memory | Uses MBs of memory |
| Heavier isolation | Lighter isolation |

**Docker** is the most popular tool for building and running containers. You define a **Dockerfile** with instructions, build a **container image**, and run it as a container.

## Amazon ECS (Elastic Container Service)

ECS is an AWS-native **container orchestration service**. It manages where and how your containers run across a cluster of compute resources.

### Key ECS Concepts

| Concept | What It Is |
|---------|-----------|
| **Cluster** | A group of compute resources that run your containers |
| **Task definition** | A blueprint describing your container(s) -- image, CPU, memory, ports, environment variables |
| **Task** | A running instance of a task definition (like running a Docker container) |
| **Service** | Keeps a specified number of tasks running and replaces failed ones automatically |

### Launch Types

ECS offers two ways to run your containers:

**Fargate (Serverless)**
- AWS manages all the servers
- You just specify CPU and memory for each task
- No need to provision, patch, or scale EC2 instances
- Pay per task based on vCPU and memory used

**EC2 Launch Type**
- You create and manage EC2 instances in your cluster
- Full control over instance type, OS, and configuration
- Better for workloads that need GPU access or specific instance types
- You handle capacity planning and patching

**For beginners:** Start with Fargate. It removes the overhead of managing servers so you can focus on your application.

### Amazon ECR (Elastic Container Registry)

**ECR** is a fully managed Docker container registry. You store, manage, and deploy container images from ECR to ECS or EKS. Think of it as a private repository for your container images, similar to Docker Hub but integrated with AWS IAM for security.

## Amazon EKS (Elastic Kubernetes Service)

EKS is a managed **Kubernetes** service. Kubernetes is an open-source platform for automating container deployment, scaling, and management. EKS runs the Kubernetes **control plane** for you -- the components that schedule containers, manage state, and handle API requests.

### Why EKS?

- Your team already uses Kubernetes and wants to run it on AWS
- You need to run the same workloads on AWS and on-premises (portability)
- You want access to the large Kubernetes ecosystem of tools and plugins

### EKS also supports Fargate and EC2:

| Compute Option | ECS Term | EKS Term |
|----------------|----------|----------|
| **Serverless** | Fargate task | Fargate pod |
| **Self-managed** | EC2 task | EC2 worker node |

## ECS vs. EKS

| Factor | ECS | EKS |
|--------|-----|-----|
| **Orchestration** | AWS-native | Kubernetes (open-source) |
| **Learning curve** | Lower -- simpler concepts | Higher -- Kubernetes is complex |
| **Portability** | AWS only | Multi-cloud and on-premises |
| **Ecosystem** | AWS tools and integrations | Large Kubernetes ecosystem |
| **Best for** | Teams new to containers, AWS-focused workloads | Teams with Kubernetes experience, multi-cloud needs |
| **Fargate support** | Yes | Yes |

## When to Use Containers vs. Lambda

| Factor | Containers (ECS/EKS) | Lambda |
|--------|----------------------|--------|
| **Execution time** | No limit | Max 15 minutes |
| **Control** | Full control over runtime and OS | AWS manages everything |
| **Startup** | Seconds (already running in a service) | Cold starts possible |
| **Best for** | Long-running applications, microservices | Short event-driven tasks, APIs |

## Key Takeaways

- Containers package apps with their dependencies into portable, lightweight units
- **ECS** is AWS-native and simpler to learn -- great for teams new to containers
- **EKS** runs Kubernetes on AWS -- best for teams with existing Kubernetes skills
- **Fargate** is serverless: AWS manages the servers so you focus on your containers
- **EC2 launch type** gives you full control over the compute infrastructure
- **ECR** stores your container images securely with AWS IAM integration
- Choose ECS for simplicity, EKS for portability and Kubernetes ecosystem access
