# AWS CloudTrail Auditing

**AWS CloudTrail** records every API call made in your AWS account. Whenever someone launches an EC2 instance, changes an IAM policy, or deletes an S3 bucket, CloudTrail logs **who** did it, **what** they did, **when** they did it, and **from where**. This makes CloudTrail essential for security auditing, compliance, and troubleshooting.

## What CloudTrail Records

Every action in AWS is an API call -- even clicking a button in the Console. CloudTrail captures each call as an **event** containing:

| Field | What It Tells You |
|-------|------------------|
| **userIdentity** | Who made the call (IAM user, role, or AWS service) |
| **eventTime** | When the call was made (UTC timestamp) |
| **eventName** | The API action (e.g., `RunInstances`, `DeleteBucket`) |
| **eventSource** | Which AWS service was called (e.g., `ec2.amazonaws.com`) |
| **sourceIPAddress** | The IP address of the caller |
| **requestParameters** | The input sent with the API call |
| **responseElements** | The output returned by the API |

**Example question CloudTrail answers:** "Who terminated the production EC2 instance at 3:00 AM on Tuesday?"

## Event Types

CloudTrail categorizes events into two main types:

| Type | What It Captures | Examples | Free? |
|------|-----------------|----------|-------|
| **Management events** | Control plane operations on AWS resources | Create/delete/modify resources, IAM changes, VPC changes | Yes (last 90 days) |
| **Data events** | Data plane operations on resource contents | S3 object-level activity (GetObject, PutObject), Lambda invocations | No (extra charge) |

**Management events** are logged by default. They cover actions like launching EC2 instances, creating S3 buckets, and changing security groups.

**Data events** are high-volume and disabled by default. You enable them when you need to track individual object reads/writes in S3 or individual Lambda invocations.

## Event History

CloudTrail **Event History** is available in every AWS account automatically, with no setup required.

- Stores the last **90 days** of management events
- **Free** -- no charges for viewing or searching
- Searchable by event name, resource type, user, and time range
- Downloadable as JSON or CSV

Event History is great for quick lookups, but it only keeps 90 days of data and does not include data events.

## Trails

For long-term storage and advanced analysis, you create a **trail**. A trail delivers CloudTrail events to destinations you choose:

| Destination | Purpose |
|-------------|---------|
| **Amazon S3** | Long-term storage and archival; keep logs for years |
| **CloudWatch Logs** | Real-time monitoring; set alarms on specific API calls |
| **Amazon EventBridge** | Trigger automated actions based on API events |

### Trail Options

- **Single-Region trail** -- captures events in one Region only
- **Multi-Region trail** -- captures events across all Regions (recommended)
- **Organization trail** -- captures events for all accounts in an AWS Organization

AWS delivers **one free copy** of management events to your S3 bucket. You pay only for S3 storage and any additional copies.

## CloudTrail Insights

**CloudTrail Insights** detects unusual API activity in your account. It analyzes your normal patterns and alerts you when something looks different.

**Examples of what Insights detects:**

- A sudden spike in `RunInstances` calls (someone launching many EC2 instances)
- An unusual number of `AuthorizeSecurityGroupIngress` calls (someone opening ports)
- API error rate increases (possible misconfiguration or attack)

Insights is an optional feature that runs on top of your trail data.

## CloudTrail vs. CloudWatch

These two services work together but serve different purposes:

| Aspect | CloudTrail | CloudWatch |
|--------|-----------|------------|
| **What it tracks** | API calls (who did what) | Performance metrics (how things are running) |
| **Focus** | Auditing and security | Monitoring and alerting |
| **Example** | "Who deleted the database?" | "Is the database CPU above 80%?" |
| **Output** | Event logs (JSON) | Metrics, alarms, dashboards |

**Use both together:** CloudTrail tells you who changed something. CloudWatch tells you how that change affected performance.

## Key Takeaways

- CloudTrail records every API call in your account: **who, what, when, and from where**
- **Event History** stores 90 days of management events for free with no setup
- **Trails** deliver logs to S3 for long-term storage and to CloudWatch Logs for real-time alerts
- **Management events** (control plane) are logged by default; **data events** (data plane) cost extra
- **CloudTrail Insights** detects unusual API activity patterns
- CloudTrail is for **auditing** (who did what); CloudWatch is for **monitoring** (how things perform)
