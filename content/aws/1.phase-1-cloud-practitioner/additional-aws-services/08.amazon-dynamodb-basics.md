# Amazon DynamoDB Basics

**Amazon DynamoDB** is a fully managed, serverless **NoSQL database** that delivers single-digit millisecond performance at any scale. You create a table, and DynamoDB handles provisioning, replication, patching, and scaling automatically. It powers some of the world's largest applications, including Amazon.com.

## SQL vs. NoSQL

Before diving into DynamoDB, it helps to understand how it differs from traditional relational databases like Amazon RDS.

| Feature | Relational (SQL) | DynamoDB (NoSQL) |
|---------|-----------------|------------------|
| **Data model** | Tables with fixed rows and columns | Flexible items with varying attributes |
| **Schema** | Must define schema before inserting data | Schema-less -- each item can have different attributes |
| **Joins** | Supports JOIN across tables | No JOINs -- denormalize data into single tables |
| **Scaling** | Vertical (bigger server) | Horizontal (automatic partitioning) |
| **Best for** | Complex queries with relationships | High-speed reads/writes at massive scale |

**When to choose DynamoDB:** You need fast, predictable performance for large volumes of data with simple access patterns (get by key, query by key + sort).

**When to choose RDS:** You need complex queries with joins across multiple tables and strict data relationships.

## Data Model

### Tables, Items, and Attributes

DynamoDB organizes data into three levels:

| Concept | SQL Equivalent | Description |
|---------|---------------|-------------|
| **Table** | Table | A collection of data (e.g., `Users`, `Orders`) |
| **Item** | Row | A single record in the table (e.g., one user) |
| **Attribute** | Column | A data field on an item (e.g., `name`, `email`) |

The key difference: in SQL, every row must have the same columns. In DynamoDB, each item can have **different attributes**. One user might have a `phone` attribute while another does not.

### Primary Keys

Every DynamoDB table needs a **primary key** to identify items uniquely. Two options:

**Simple primary key (partition key only)**

```
Table: Users
Partition Key: userId
─────────────────────────
| userId  | name    | email           |
| "u-001" | "Alice" | "alice@test.com"|
| "u-002" | "Bob"   | "bob@test.com"  |
```

DynamoDB hashes the partition key to decide which internal partition stores the item.

**Composite primary key (partition key + sort key)**

```
Table: Orders
Partition Key: userId    Sort Key: orderDate
──────────────────────────────────────────────
| userId  | orderDate    | total  | status   |
| "u-001" | "2025-01-15" | 49.99  | "shipped"|
| "u-001" | "2025-02-20" | 29.99  | "pending"|
| "u-002" | "2025-01-10" | 89.99  | "shipped"|
```

The partition key groups related items together. The sort key orders them within the group. This lets you query all orders for a specific user, sorted by date.

## Capacity Modes

DynamoDB offers two ways to handle read and write throughput:

| Mode | How It Works | Best For |
|------|-------------|----------|
| **On-Demand** | Scales automatically; pay per request | Unpredictable or new workloads |
| **Provisioned** | You set read/write capacity units in advance | Predictable, steady workloads (cheaper) |

**On-demand** is the simplest option -- DynamoDB handles all scaling and you pay only for what you use. No capacity planning needed.

**Provisioned** costs less for consistent workloads. You specify how many read and write capacity units you need. You can also enable **Auto Scaling** to adjust provisioned capacity automatically based on actual usage.

You can switch between modes once every 24 hours.

## Key Features

### High Availability

DynamoDB automatically replicates data across **three Availability Zones** in a Region. This provides 99.99% availability (99.999% with Global Tables).

### Global Tables

**Global Tables** replicate your table across multiple AWS Regions for low-latency access worldwide. Changes in one Region automatically propagate to all other Regions.

### DynamoDB Streams

**Streams** capture a time-ordered log of changes to your table (inserts, updates, deletes). You can trigger a Lambda function whenever data changes -- useful for real-time processing, analytics, or keeping other systems in sync.

### DynamoDB Accelerator (DAX)

**DAX** is a fully managed in-memory cache that sits in front of DynamoDB. It reduces read response times from milliseconds to **microseconds**. Your application reads from DAX first; if the data is not cached, DAX fetches it from DynamoDB.

### Backup and Recovery

- **On-demand backups** -- create full backups at any time
- **Point-in-time recovery (PITR)** -- restore your table to any second in the last 35 days
- **Encryption at rest** -- enabled by default using AWS-managed keys

## Free Tier

DynamoDB's free tier includes:

- **25 GB** of storage
- **25 read capacity units** and **25 write capacity units** (provisioned mode)
- Enough for approximately **200 million requests per month**

The free tier is available permanently, not just for 12 months.

## Key Takeaways

- DynamoDB is a serverless NoSQL database with single-digit millisecond performance at any scale
- Data is organized into **tables**, **items** (rows), and **attributes** (columns) with a flexible schema
- Every table needs a **primary key** -- either a partition key alone or a partition key plus sort key
- **On-demand mode** scales automatically with no planning; **provisioned mode** costs less for steady workloads
- Data is replicated across **three AZs** automatically; **Global Tables** extend this across Regions
- **DAX** adds an in-memory cache for microsecond reads; **Streams** trigger actions on data changes
- DynamoDB is ideal for high-throughput, low-latency workloads with simple access patterns
