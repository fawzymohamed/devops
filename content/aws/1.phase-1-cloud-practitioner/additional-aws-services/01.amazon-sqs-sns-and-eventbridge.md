# Amazon SQS, SNS & EventBridge

AWS offers three messaging services that help you build **loosely coupled, event-driven applications**. Each solves a different communication problem: SQS queues messages for one consumer, SNS broadcasts messages to many subscribers, and EventBridge routes events based on rules.

## Amazon SQS (Simple Queue Service)

SQS is a fully managed **message queue**. It acts as a buffer between a producer (the component that sends messages) and a consumer (the component that processes them). If the consumer is slow or offline, messages wait safely in the queue.

### How SQS Works

1. **Producer** sends a message to the queue
2. Message is stored across multiple servers for durability
3. **Consumer** polls the queue and retrieves the message
4. Consumer processes the message and deletes it from the queue

If the consumer fails to process, the message becomes visible again for retry.

### Queue Types

| Feature | Standard Queue | FIFO Queue |
|---------|---------------|------------|
| **Ordering** | Best-effort (may arrive out of order) | Exact order guaranteed |
| **Delivery** | At least once (possible duplicates) | Exactly once |
| **Throughput** | Nearly unlimited | Up to 3,000 messages/sec with batching |
| **Use case** | High-throughput tasks like log processing | Financial transactions, order processing |

### Key Features

- **Visibility timeout** -- hides a message while a consumer processes it (default 30 seconds)
- **Dead-letter queue** -- captures messages that fail processing after multiple attempts
- **Retention** -- messages stay in the queue for up to 14 days (default 4 days)

## Amazon SNS (Simple Notification Service)

SNS is a fully managed **publish/subscribe (pub/sub)** service. Instead of one consumer, SNS sends a message to **all subscribers** at once.

### How SNS Works

1. You create a **topic** (a communication channel)
2. **Subscribers** register for the topic (email, SMS, Lambda, SQS, HTTP)
3. A **publisher** sends a message to the topic
4. SNS delivers the message to every subscriber simultaneously

### Common Subscriber Types

| Subscriber | Example Use |
|------------|-------------|
| **Email** | Alert a team about system errors |
| **SMS** | Send verification codes to users |
| **Lambda** | Trigger a function to process the event |
| **SQS** | Fan out to multiple queues for parallel processing |
| **HTTP/HTTPS** | Notify an external webhook |

**SNS + SQS pattern:** Publish one message to SNS, which fans it out to multiple SQS queues. Each queue feeds a different processing system. This is called the **fan-out pattern**.

## Amazon EventBridge

EventBridge is a **serverless event bus** that routes events from many sources to targets based on rules you define.

### How EventBridge Works

1. An **event source** emits an event (AWS service, your app, or a SaaS provider)
2. EventBridge matches the event against your **rules**
3. Matching events are sent to one or more **targets** (Lambda, SQS, Step Functions, etc.)

### What Makes EventBridge Different

- **Schema registry** -- automatically discovers and stores event formats so you know what data to expect
- **Content-based filtering** -- route events using detailed rules that match fields inside the event payload
- **SaaS integration** -- receives events from third-party providers like Shopify, Zendesk, and Datadog
- **Scheduling** -- run tasks on a schedule (like cron jobs) without extra infrastructure

## Choosing the Right Service

| Scenario | Best Service | Why |
|----------|-------------|-----|
| Buffer messages between two services | **SQS** | Queue holds messages until the consumer is ready |
| Send one alert to many recipients | **SNS** | Pub/sub pushes to all subscribers at once |
| Route events from many sources by rules | **EventBridge** | Content-based filtering and SaaS integration |
| Fan out one event to multiple queues | **SNS + SQS** | SNS broadcasts, each SQS queue processes independently |
| Schedule a task to run every hour | **EventBridge** | Built-in scheduler, no extra setup |

## Key Takeaways

- **SQS** stores messages in a queue for one consumer -- great for decoupling and buffering
- **SNS** pushes messages to many subscribers at once using a pub/sub model
- **EventBridge** routes events from AWS, your apps, and SaaS providers using content-based rules
- Use **SQS** when you need reliable, ordered message processing between two components
- Use **SNS** when one event needs to reach multiple targets simultaneously
- Use **EventBridge** when you need intelligent routing from many event sources
