# DevOps as SDLC Evolution

The previous lessons covered Waterfall, Agile, Scrum, and Kanban. Each model improved how teams plan and build software. But even Agile teams hit a wall: they could build features fast, yet those features sat in a queue waiting weeks or months to reach production. The bottleneck was not development -- it was the handoff between development and operations.

DevOps is the answer to that bottleneck. It is not a replacement for earlier models. It is the next step in the evolution of how we deliver software. As a DevOps architect, understanding this evolution helps you see why DevOps practices exist and how they fit together.

## The Problem: The Wall Between Dev and Ops

In traditional organizations, development teams and operations teams worked in separate silos. Each had different goals:

| Team | Goal | Incentive |
|------|------|-----------|
| **Development** | Ship new features fast | Change the system |
| **Operations** | Keep production stable | Prevent changes |

These goals directly conflict. Developers wanted to release often. Operations wanted to release rarely. The result was a painful handoff process:

1. Developers wrote code and "threw it over the wall" to operations
2. Operations received unfamiliar code with little context
3. Deployments failed, causing outages and blame
4. Both teams became defensive and risk-averse
5. Release cycles stretched to months or even quarters

Agile made development faster, but it made this wall more painful. Teams could build features in two-week Sprints, but then waited six weeks for a deployment window. The faster development moved, the bigger the pile of unreleased work grew.

DevOps did not appear out of nowhere. It grew from ideas that built on each other over decades.

**2008 -- The Agile Infrastructure Thread.** At the Agile Conference in Toronto, Patrick Debois and Andrew Shafer discussed applying Agile ideas to infrastructure work. This planted the seed.

**2009 -- DevOpsDays.** Patrick Debois organized the first DevOpsDays conference in Ghent, Belgium. The name "DevOps" was born. It combined "Development" and "Operations" into a single word -- and a single mission.

**2009-2013 -- Core Practices Emerge.** Companies like Flickr, Etsy, and Netflix showed that deploying many times per day was possible and safe. John Allspaw and Paul Hammond gave the famous talk "10+ Deploys Per Day" at Flickr, showing how dev and ops could work as one team.

**2013-Present -- Mainstream Adoption.** Tools like Docker, Kubernetes, Jenkins, and Terraform made DevOps practices accessible to every team. The "State of DevOps" reports provided data showing that DevOps teams deliver faster, more reliably, and with fewer failures.

## What is the DevOps Lifecycle?

Unlike Waterfall's straight line or Scrum's Sprints, the DevOps lifecycle is a **continuous loop**. Software is planned, built, tested, released, operated, and monitored -- and the feedback from monitoring drives the next round of planning.

Each stage feeds into the next, and monitoring feeds back into planning. There is no "done" -- only continuous improvement. This is the fundamental shift from traditional SDLC thinking.

## Core DevOps Practices

The DevOps lifecycle is powered by a set of practices. Each one removes a bottleneck from the traditional handoff process.

### Continuous Integration (CI)

Developers merge their code into a shared repository multiple times per day. Every merge triggers automated builds and tests. This catches problems early, while the code is fresh in the developer's mind.

### Continuous Delivery (CD)

Every code change that passes CI is automatically prepared for release. The software is always in a deployable state. Teams can release at any time with the push of a button -- or automatically.

### Infrastructure as Code (IaC)

Servers, networks, and cloud resources are defined in code files, not set up by hand. This means infrastructure can be versioned, reviewed, tested, and replicated just like application code.

### Continuous Monitoring

Production systems are watched around the clock with automated tools. Metrics, logs, and alerts tell the team how the software behaves in the real world. This feedback drives the next cycle of improvements.

### Shift-Left

Testing, security, and quality checks move earlier ("left") in the lifecycle. Instead of finding bugs after deployment, teams find them during coding. Instead of running security scans before release, teams scan during every build.

DevOps does not throw away everything from Waterfall or Agile. It absorbs their best ideas:

- **From Waterfall**: Structured phases and clear documentation when needed
- **From Agile**: Short iterations, customer feedback, and adaptability
- **From Lean/Kanban**: Flow optimization, waste removal, and WIP limits
- **New in DevOps**: Automation, shared ownership, and continuous everything

## Common Mistakes to Avoid

1. **Treating DevOps as just tools** -- Buying CI/CD tools does not make a team DevOps. DevOps is a culture change first. Tools support the culture, but they cannot create it. A team that automates a broken process still has a broken process.

2. **Creating a "DevOps team" silo** -- Some organizations create a separate DevOps team and call it done. This just moves the wall instead of removing it. DevOps means development and operations work **together**, not that a third team sits between them.

3. **Skipping the culture shift** -- DevOps needs blameless post-mortems, shared responsibility, and trust between teams. Without these cultural changes, automation alone will not fix the Dev/Ops divide.

## Key Takeaways

- DevOps evolved from Waterfall, Agile, and Lean to solve the gap between development and operations
- The core problem was conflicting goals: Dev wanted change, Ops wanted stability
- Patrick Debois coined the term "DevOps" at the first DevOpsDays in 2009
- The DevOps lifecycle is a continuous loop: Plan, Code, Build, Test, Release, Deploy, Operate, Monitor
- Key practices include CI/CD, Infrastructure as Code, continuous monitoring, and shift-left
- DevOps is a culture change supported by automation -- not just a set of tools
