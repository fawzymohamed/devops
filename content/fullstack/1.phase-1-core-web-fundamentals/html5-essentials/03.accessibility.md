# Web Accessibility with HTML5 & ARIA

Web accessibility means building websites that everyone can use, including people with disabilities. About 15% of the world's population has some form of disability - visual, auditory, motor, or cognitive. Accessible websites work for all users, regardless of how they navigate the web.

In this lesson, you'll learn how to build accessible websites using semantic HTML, ARIA attributes, keyboard navigation, and best practices that make the web inclusive for everyone.

## What Is Web Accessibility?

**Web accessibility (a11y)** ensures that people with disabilities can:
- Perceive content (see or hear it)
- Navigate the interface (keyboard, screen reader, voice control)
- Understand the content and interface
- Interact with forms, buttons, and controls

**Who Benefits:**
- **Blind users** using screen readers
- **Low vision users** using screen magnifiers
- **Deaf users** needing captions or transcripts
- **Motor impaired users** using keyboard or voice control
- **Cognitive disability users** needing clear language and structure
- **Everyone** in situational contexts (bright sunlight, broken mouse, slow connection)

## Semantic HTML: The Foundation

**Semantic HTML is 80% of accessibility.** When you use the right HTML elements, you get accessibility features for free.

### Semantic vs. Non-Semantic Buttons

**Bad (Not Accessible):**
```html
<!-- Looks like a button, but screen readers treat it as text -->
<div class="button" onclick="submitForm()">
  Submit
</div>
```

**Problems:**
- Not keyboard accessible (can't press Enter/Space)
- No focus indicator
- Screen readers don't announce it as a button
- No built-in click behavior

**Good (Accessible):**
```html
<!-- Semantic button with built-in accessibility -->
<button type="button" onclick="submitForm()">
  Submit
</button>
```

**Benefits:**
- Keyboard accessible (Enter/Space)
- Focusable with Tab key
- Screen readers announce "Submit, button"
- Built-in click behavior

### Semantic Landmarks

Use semantic HTML5 elements to create page landmarks that screen readers navigate:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Accessible Page Structure</title>
</head>
<body>
  <!-- Header landmark -->
  <header>
    <h1>Site Name</h1>
    <nav aria-label="Main Navigation">
      <a href="/">Home</a>
      <a href="/about">About</a>
      <a href="/contact">Contact</a>
    </nav>
  </header>

  <!-- Main content landmark -->
  <main>
    <h1>Page Title</h1>
    <p>Main content goes here...</p>
  </main>

  <!-- Complementary content landmark -->
  <aside>
    <h2>Related Links</h2>
    <ul>
      <li><a href="/article1">Article 1</a></li>
      <li><a href="/article2">Article 2</a></li>
    </ul>
  </aside>

  <!-- Footer landmark -->
  <footer>
    <p>&copy; 2026 Company Name</p>
  </footer>
</body>
</html>
```

**Screen Reader Benefits:**
- Users can jump between landmarks (header, nav, main, aside, footer)
- Provides a mental map of the page structure
- Allows skipping to main content quickly

## ARIA: Accessible Rich Internet Applications

**ARIA** adds accessibility information when semantic HTML isn't enough. It provides three main types of attributes:

1. **Roles** - What something is (button, dialog, alert)
2. **Properties** - What something does (aria-label, aria-required)
3. **States** - Current condition (aria-expanded, aria-checked)

**First Rule of ARIA:** Don't use ARIA if semantic HTML works.

### ARIA Roles

Roles define what an element is or does.

**Example - Custom Navigation:**
```html
<!-- Semantic HTML (preferred) -->
<nav>
  <a href="/">Home</a>
  <a href="/about">About</a>
</nav>

<!-- ARIA role (if you can't use <nav>) -->
<div role="navigation">
  <a href="/">Home</a>
  <a href="/about">About</a>
</div>
```

**Common ARIA Roles:**
- `role="navigation"` - Navigation section (use `<nav>` instead)
- `role="main"` - Main content (use `<main>` instead)
- `role="banner"` - Site header (use `<header>` instead)
- `role="contentinfo"` - Site footer (use `<footer>` instead)
- `role="search"` - Search form
- `role="alert"` - Important message
- `role="dialog"` - Modal dialog
- `role="button"` - Button (use `<button>` instead)

### ARIA Labels

ARIA labels provide accessible names for elements.

#### aria-label

Provides an invisible label for screen readers:

```html
<!-- Icon-only button needs a label -->
<button aria-label="Close dialog">
  ‚úï
</button>

<!-- Search button with icon -->
<button type="submit" aria-label="Search">
  <svg><!-- magnifying glass icon --></svg>
</button>

<!-- Social media links -->
<a href="https://twitter.com/username" aria-label="Follow us on Twitter">
  <svg><!-- Twitter icon --></svg>
</a>
```

**When to Use:**
- Icon-only buttons
- Graphical links
- When visible text doesn't fully explain the element's purpose

#### aria-labelledby

Points to another element that contains the label:

```html
<section aria-labelledby="products-heading">
  <h2 id="products-heading">Our Products</h2>
  <p>Product list...</p>
</section>

<!-- Modal dialog -->
<div role="dialog" aria-labelledby="dialog-title">
  <h2 id="dialog-title">Confirm Delete</h2>
  <p>Are you sure?</p>
  <button>Yes</button>
  <button>No</button>
</div>
```

#### aria-describedby

Points to additional descriptive text:

```html
<label for="password">Password:</label>
<input
  type="password"
  id="password"
  aria-describedby="password-hint"
>
<p id="password-hint">
  Must be at least 8 characters with one number.
</p>

<!-- Error message -->
<label for="email">Email:</label>
<input
  type="email"
  id="email"
  aria-describedby="email-error"
  aria-invalid="true"
>
<p id="email-error" role="alert">
  Please enter a valid email address.
</p>
```

### ARIA States and Properties

Communicate dynamic state changes to assistive technologies.

#### aria-expanded

Indicates if a collapsible element is open or closed:

```html
<button
  aria-expanded="false"
  aria-controls="menu"
  onclick="toggleMenu()"
>
  Menu
</button>

<ul id="menu" hidden>
  <li><a href="/">Home</a></li>
  <li><a href="/about">About</a></li>
  <li><a href="/contact">Contact</a></li>
</ul>
```

When clicked, JavaScript updates `aria-expanded` to `"true"` and removes `hidden`.

#### aria-hidden

Hides content from screen readers:

```html
<!-- Decorative icon (no meaning) -->
<span aria-hidden="true">üé®</span>
<span>Color Picker</span>

<!-- Duplicate content hidden for screen readers -->
<div>
  <h2>Contact Us</h2>
  <span aria-hidden="true">Contact Us</span> <!-- Hidden duplicate -->
</div>
```

**Warning:** Don't hide focusable elements with `aria-hidden="true"`.

#### aria-live

Announces dynamic content changes:

```html
<!-- Polite announcement (waits for user to pause) -->
<div aria-live="polite" role="status">
  5 new messages
</div>

<!-- Assertive announcement (interrupts immediately) -->
<div aria-live="assertive" role="alert">
  Error: Your session has expired
</div>
```

**Live Region Politeness:**
- `polite` - Wait until user pauses (status updates, notifications)
- `assertive` - Interrupt immediately (errors, critical alerts)
- `off` - Don't announce (default)

#### aria-current

Indicates the current item in navigation:

```html
<nav>
  <a href="/">Home</a>
  <a href="/about" aria-current="page">About</a> <!-- Current page -->
  <a href="/contact">Contact</a>
</nav>
```

#### aria-required

Indicates required form fields:

```html
<!-- Semantic HTML (preferred) -->
<input type="text" required>

<!-- ARIA (if you can't use required attribute) -->
<input type="text" aria-required="true">
```

## Keyboard Accessibility

Many users navigate without a mouse. Your site must work with keyboard alone.

### Focus Management

Ensure all interactive elements are keyboard accessible:

```html
<!-- Good: Native button is focusable -->
<button>Click Me</button>

<!-- Good: Link is focusable -->
<a href="/page">Go to Page</a>

<!-- Bad: Div is not focusable by default -->
<div onclick="doSomething()">Click Me</div>

<!-- Fix: Add tabindex -->
<div tabindex="0" role="button" onclick="doSomething()">
  Click Me
</div>
```

### Tabindex Values

- `tabindex="0"` - Element is focusable in natural tab order
- `tabindex="-1"` - Element is focusable programmatically (via JavaScript) but not via Tab key
- `tabindex="1+"` - Custom tab order (avoid - breaks natural flow)

**Example - Skip to Main Content Link:**
```html
<body>
  <!-- Skip link appears on focus -->
  <a href="#main" class="skip-link">Skip to main content</a>

  <header>
    <!-- Many navigation links here -->
  </header>

  <main id="main" tabindex="-1">
    <h1>Main Content</h1>
    <p>Content...</p>
  </main>
</body>
```

### Visible Focus Indicators

Never remove focus outlines completely:

```css
/* Bad - removes focus indicator */
button:focus {
  outline: none;
}

/* Good - custom focus indicator */
button:focus {
  outline: 2px solid #3b82f6;
  outline-offset: 2px;
}

/* Modern approach - focus-visible */
button:focus-visible {
  outline: 2px solid #3b82f6;
  outline-offset: 2px;
}
```

**`:focus-visible`** shows focus only for keyboard users, not mouse clicks.

### Keyboard Event Handling

If you use `onclick`, support keyboard events too:

```html
<!-- Button - Enter/Space work automatically -->
<button onclick="doSomething()">Click</button>

<!-- Custom element - add keyboard handler -->
<div
  role="button"
  tabindex="0"
  onclick="doSomething()"
  onkeydown="handleKeydown(event)"
>
  Custom Button
</div>

<script>
function handleKeydown(event) {
  // Activate on Enter or Space
  if (event.key === 'Enter' || event.key === ' ') {
    event.preventDefault();
    doSomething();
  }
}
</script>
```

## Images and Alt Text

All images need alternative text for screen readers.

### Informative Images

Describe the content or function:

```html
<!-- Content image -->
<img src="chart.png" alt="Bar chart showing sales increased 25% in Q4">

<!-- Logo link -->
<a href="/">
  <img src="logo.png" alt="Company Name - Home">
</a>

<!-- Icon button -->
<button>
  <img src="trash-icon.svg" alt="Delete">
</button>
```

### Decorative Images

Use empty alt for images that don't convey information:

```html
<!-- Decorative border image -->
<img src="decorative-border.png" alt="">

<!-- Icon with adjacent text -->
<button>
  <img src="save-icon.svg" alt="">
  Save Document
</button>
```

**Rule:** If removing the image doesn't affect understanding, use `alt=""`.

### Complex Images

Provide detailed descriptions:

```html
<figure>
  <img
    src="complex-chart.png"
    alt="Sales data chart"
    aria-describedby="chart-description"
  >
  <figcaption id="chart-description">
    This chart shows quarterly sales from 2020 to 2025.
    Sales started at $1M in Q1 2020 and grew to $5M by Q4 2025,
    with the fastest growth occurring in 2023.
  </figcaption>
</figure>
```

## Accessible Forms

Forms must be accessible to everyone. Follow these practices:

### Label Every Input

```html
<!-- Good: Label connected with 'for' -->
<label for="email">Email Address:</label>
<input type="email" id="email" name="email" required>

<!-- Alternative: Wrap input in label -->
<label>
  Email Address:
  <input type="email" name="email" required>
</label>
```

### Group Related Fields

```html
<fieldset>
  <legend>Contact Information</legend>

  <label for="email">Email:</label>
  <input type="email" id="email" name="email">

  <label for="phone">Phone:</label>
  <input type="tel" id="phone" name="phone">
</fieldset>
```

### Error Messages

Link error messages to inputs:

```html
<label for="username">Username:</label>
<input
  type="text"
  id="username"
  name="username"
  aria-describedby="username-error"
  aria-invalid="true"
>
<p id="username-error" role="alert">
  Username must be at least 3 characters.
</p>
```

## Accessible Color and Contrast

Ensure sufficient color contrast for readability:

- **Normal text:** 4.5:1 contrast ratio minimum
- **Large text (18pt+ or 14pt+ bold):** 3:1 minimum
- **Don't rely on color alone** to convey information

**Good:**
```html
<!-- Text + icon for status -->
<p>
  <span style="color: green;">‚úì</span> Success: Your changes were saved.
</p>

<p>
  <span style="color: red;">‚úï</span> Error: Invalid email address.
</p>
```

**Bad:**
```html
<!-- Color only (colorblind users can't distinguish) -->
<p style="color: green;">Your changes were saved.</p>
<p style="color: red;">Invalid email address.</p>
```

- **Mobile:** VoiceOver (iOS), TalkBack (Android)

## Common Mistakes to Avoid

1. **Using divs as buttons**
   - Use `<button>` for clickable elements
   - Divs aren't keyboard accessible by default

2. **Removing focus outlines**
   - Keyboard users need to see where focus is
   - Customize, don't remove

3. **Empty links or buttons**
   - Every link/button needs text or `aria-label`
   - Icon-only elements must have accessible names

4. **Incorrect heading hierarchy**
   - Don't skip levels (h1 ‚Üí h2 ‚Üí h3, not h1 ‚Üí h3)
   - Use headings for structure, not styling

5. **Low color contrast**
   - Test with contrast checkers
   - Ensure 4.5:1 ratio for normal text

## Key Takeaways

- **Web accessibility** ensures websites work for everyone, including people with disabilities
- **Semantic HTML** provides 80% of accessibility automatically
- **ARIA attributes** add accessibility information when HTML isn't enough
- **First rule of ARIA:** Don't use ARIA if semantic HTML works
- **Keyboard accessibility** is essential - all interactive elements must be focusable
- **Alt text** describes image content for screen readers and when images fail to load
- **Focus indicators** must be visible for keyboard users
- **Form labels** must be connected to inputs with `for` and `id`
- **Color contrast** must meet WCAG standards (4.5:1 for normal text)
- **Testing** requires both automated tools and manual keyboard/screen reader testing

Accessibility isn't an add-on - it's a fundamental part of web development. Build accessibility in from the start, and you'll create better experiences for everyone.
